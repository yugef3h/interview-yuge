# Agent 原理、架构与工程实践

## 一、控制流：Agent 核心循环
感知·决策·行动·反馈 - 四个阶段持续循环

1.  **Perceive（感知）**：读取历史消息（`Read messages history`），获取当前上下文信息。
2.  **Decide（决策）**：调用大模型进行推理（`LLM inference`），决定下一步行动。
3.  **Act（行动）**：执行工具调用（`Dispatch tool_use calls`）。
4.  **Feedback（反馈）**：将工具执行结果注入（`Inject tool_result`），推送到下一轮循环。

**终止条件**：直到模型返回纯文本结果为止。
> 主循环不承担状态管理，也不承担工具编排。

## 二、Workflow vs Agent：核心区别对比
控制权掌握在谁手里

| 维度 | WORKFLOW | AGENT |
| :--- | :--- | :--- |
| **控制权** | 代码预定义 - 同输入同路径 | LLM 动态决策 - 可能需要评测验证 |
| **执行方式** | 工具顺序固定 - 错误走预设分支 | 工具按需选择 - 模型可自我修复 |
| **状态与记忆** | 显式状态机 - 节点跳转清晰 | 隐式上下文 - 状态在对话里累积 |
| **维护成本** | 改流程要改代码并重新部署 | 调整系统提示即可 - 无需重部署 |
| **可观测性** | 日志定位节点 - 延迟可预估 | 需完整 Trace 理解决策链 |
| **适用场景** | 流程固定、输入边界清晰 | 需要中间推理与灵活判断 |

> 很多挂着 Agent 招牌的产品，深入看更接近 Workflow。
