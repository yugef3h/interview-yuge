## 〇、2026 架构师面试的范式转移（新增）

### 0.1 面试本身的变革

2026 年的架构师面试已经发生了根本性变化。**92% 的开发者每日使用 AI 工具，约 41% 的代码为 AI 生成**（DORA 2026），面试不再考察"你会不会写代码"，而是考察**判断力、架构思维、系统级思考**。

| 变革维度 | 2023-2024 | 2026 |
|----------|-----------|------|
| **编码轮** | 手写算法/数据结构 | Meta 率先引入 **AI-Enabled Round**：60min CoderPad，可自由使用 GPT-5/Claude/Gemini。考察：策略性 Prompt、识别 AI 错误、将 AI 输出精炼为生产代码 |
| **系统设计** | 高级岗才有 | **从 L4/中级开始必考**。因为 AI 能生成实现代码，区分度在系统思维 |
| **Code Review** | 次要环节 | **比 Code Writing 更重要**。审查 AI 生成代码中的竞态条件、安全漏洞、Prompt 注入 |
| **行为轮** | 团队协作、冲突处理 | 新增六维度：架构设计、AI 辅助执行、系统与运维、产品判断、学习速度、AI 文化 |
| **评估标准** | 方案正确性 | **工程化落地能力**：能否设计"不崩的系统"，而非"知道这个概念" |

> **核心信号**：面试官不再问"你知不知道 X"，而是问"你能不能设计一个在生产环境规模化运行且不崩的 X 系统"。

### 0.2 从"Coder"到"Orchestrator"的角色重构

2026 年架构师的核心身份已从 **代码创作者** 转变为 **AI 编排者**：

| 旧范式 | 新范式（2026） |
|--------|---------------|
| 手动编写代码 | 策略性 AI 编排（多模型、多 Agent、多工具链） |
| 语法精通 | 系统思维 + 架构权衡 |
| 写代码行数 | 验证 AI 输出、审查安全与正确性 |
| 工具使用者 | 工具编排者（Agent-Ops） |
| Prompt Engineering 1.0（聊天指令） | Prompt Engineering 2.0（结构化决策树、多模态上下文、意图规格说明） |

**新角色分化**：
- **AI Orchestrator / Agent-Ops**：管理多 Agent 系统、目标对齐、可观测性
- **AI Integrator**：将 AI Agent 嵌入遗留企业基础设施
- **Code Reviewer 2.0**：审计 AI 生成提交的安全性、检测幻觉库引用
- **AI Automation Engineer**：桥接 RPA 与 LLM 驱动的 Agentic 系统（$135K–$200K+）

### 2.3 企业级 RAG 架构（最高频系统设计，2026 全面升级）

**标准链路**（需能画数据流 + 控制流）：  
接入 → 解析/OCR → 清洗 → 分块 → Embedding → **分片索引** → 元数据/权限 → 增量更新 → Query 管线 → 混合召回 → 融合/Rerank → **Context Assembly** → 生成 → 溯源 → 评测回流

| 子题 | 架构师关注点（2026 升级） | 典型追问 |
|------|--------------------------|----------|
| **千万级文档** | 分片策略、冷热分层、并行索引、检索扇出 | 单租户 10 亿 chunk 怎么扩？重建索引如何不停机？ |
| **多租户** | metadata 隔离、独立 rerank 配额、审计 | 共享索引 vs 独立索引的 TCO？**共享 vLLM + 每租户独立 Collection（推荐）** |
| **权限** | ABAC/行级 filter、检索前过滤 vs 后过滤 | 误杀召回 vs 越权泄露，优先防哪个？**先防泄露，再用审计发现误杀** |
| **知识 SLA** | 版本号、增量、缓存失效、对账 | 法务要求 24h 内生效，架构怎么保证？ |
| **评测平台** | golden set、门禁、Bad Case 工单 | 谁有权上线新 Embedding？回归不过怎么办？**Eval-as-Code：PR 触发评测、不过自动阻断** |
| **成本** | Embedding 批处理、Rerank Top-N、缓存 | Token+向量+GPU 各占多少？先砍哪块？**自托管 H100：$59/百万查询 vs 托管 API：$610/百万查询** |
| **（新）性能基准** | p50/p99 延迟分解 | Embedding(2-5ms GPU vs 400ms API) + Vector Search(1-3ms vs 250ms) + LLM TTFT(30-80ms vs 1500ms) |
| **（新）Context Assembly** | Context Infrastructure Stack 四层 | 检索到的不等于该用的——Reasoning 层解决多源冲突（Slack 说 retry=3，Confluence 说 5） |

**面经锚点**：飞书2面（合同 RAG 全链路）、淘宝（RRF/HNSW/评估）、字节1面（索引）、腾讯2面（召回/TTFT）

**2026 新增：RAG 性能基准案例**（Spheron 2026 生产数据）：
- 自托管 H100 裸金属：p50 95ms, p99 190ms, 月成本 $15,200（2M 查询/天）
- 托管 API 栈：p50 920ms, p99 1800ms, 月成本 $47,000
- **结论**：月查询量 >50 万时，自托管 GPU 开始具备经济优势

### 2.4 Agent / 多智能体编排平台（2026 最大考点迁移）

| 主题 | 架构师答法（2026 全面升级） | 面经锚点 |
|------|---------------------------|----------|
| **Runtime 五模块** | Loop、Tool Registry、State Store、Trace、Policy Engine | 豆包总结「自研五模块」推演 |
| **中控调度** | Manager-Worker；子 Agent 禁止横向通信；**新增 Subagents / Agent Teams / Agent Swarms 三种模式的选型矩阵** | 字节2面、腾讯&字节 |
| **状态持久化** | Checkpoint 存哪（Redis vs DB）；会话与任务分离 | 腾讯 Redis、字节 LangGraph |
| **长任务** | 目标锚定、步数上限、摘要压缩、断点续跑 | 豆包总结模块三 |
| **百级工具** | 分层路由 + 向量预筛 + 服务端分发；**MCP Server 动态发现替代硬编码** | 字节/腾讯共识 |
| **MCP（2026 必考）** | **"AI 的 USB-C 接口"**：基于 JSON-RPC 2.0 的开放协议。Tools/Resources/Prompts 三原语。**MCP vs 传统 Function Call：标准化、动态发现、统一治理** | 快手1面、淘宝 |
| **A2A（2026 新增）** | Agent-to-Agent 水平通信协议。**MCP=垂直（Agent→工具），A2A=水平（Agent→Agent）**，分层协作非竞争 | 2026 新趋势 |
| **安全五层** | 护栏、白名单、审计、HITL、隔离；**新增 MCP 三大攻击面：Context Manipulation / Server-Side Injection / Cross-Server Compromise** | 腾讯&字节原题 |
| **死循环防御** | **三层机制：工具层硬隔离（白名单+沙箱+超时）、推理层熔断（步数上限+重复检测）、规划层自修正（Reflection Pattern）** | 2026 大厂高频 |
| **Plan-and-Execute** | **优于纯 ReAct：Planner→Executor(并行)→Re-planner→Finalizer。独立步骤并行执行，减少 Token 消耗** | 2026 新范式 |
| **Sandboxing** | **Firecracker/gVisor 隔离，"用 Docker"不够——需说明网络/文件系统边界和威胁模型** | 2026 新增考察维度 |

**连环追问（2026 升级）**：
1. 设计「Agent 任务编排系统」：DAG、重试、HITL、幂等、计费（豆包收集系统设计 #3）
2. Multi-Agent ROI：成本翻倍、效果 +10%，架构上何时值得？（资深推演 #8 升级版）
3. **（2026 新增）Agent 调了三个工具就死循环了，异常处理在哪写的？** → 三层防御 + Token 预算熔断
4. **（2026 新增）MCP Server 被攻破后横向移动到其他 Server，架构怎么防？** → 最小权限 + mTLS + 审计日志 + Cross-Server 隔离

### 2.5 大模型推理与服务化架构（2026 GPU 调度深度升级）

| 主题 | 要点（2026 升级） |
|------|------------------|
| **推理平台** | 多模型路由、队列、Continuous Batching、GPU 池化、灰度、**控制平面与推理平面分离（Fargate+vLLM scale-to-zero）** |
| **指标** | TTFT、TPS、P99、显存、$/1K tokens、**MFU（Model FLOPs Utilization）** |
| **KV Cache** | 长文本瓶颈、INT8/AWQ 取舍、GQA、**Prefix Caching、TurboQuant 压缩、FP8 KV Cache** |
| **量化** | **AWQ 4-bit（vLLM 生产首选）vs GGUF（mmap 即时映射、CPU/移动端可移植）** |
| **投机解码** | **小"草稿模型"快速预测，大模型验证，推理速度 2-5x，质量无损** |
| **流式 SSE** | 断线重连、幂等、防重复 Tool/重复计费 |
| **限流** | 滑动窗口 vs 固定窗口；漏桶削峰；**按租户/模型/API Key 多维配额** |
| **网关额度** | 分布式扣减、按天刷新、并发锁；**Token 级预算控制 + 硬性熔断（单次 >10 轮强制中断）** |
| **GPU 调度** | **eBPF 资源画像 + 强化学习预测未来 10min 需求 + 2s 内弹性伸缩** |
| **多租户 GPU** | ≤10租户 RTX 5090 / 10-30租户 RTX 6000 Pro / 30-100租户 RTX 6000 Pro 96GB |

**面经锚点**：智谱（部署/量化）、豆包收集推理优化全节

**2026 新增：vLLM 生产部署模式**：
```bash
# 多 GPU 实例并行 + Prefix Caching + AWQ 量化
for GPU_ID in 0 1 2 3; do
    CUDA_VISIBLE_DEVICES=$GPU_ID vllm serve Qwen/Qwen3.5-32B \
        --quantization awq \
        --max-model-len 16384 \
        --gpu-memory-utilization 0.85 \
        --enable-prefix-caching \
        --port $((8000 + GPU_ID)) &
done
```

**VRAM 预算公式**：模型权重 + KV Cache + FAISS 索引 + MoE 中间缓冲 + CUDA Graph 开销

## 三、2026 新增专题：Context Engineering & AI-Native 运营模型

### 3.0.1 Context Engineering——2026 架构师的第一性原理

**Context Engineering 已取代 Prompt Engineering 成为架构师核心能力。** Prompt 是指令，Context 是决定 Agent 输出正确性的全部信息集合（代码、文档、决策、策略）。

**Context Infrastructure Stack（四层模型）**：

| 层 | 职责 | 防止的失败模式 | 架构决策 |
|----|------|---------------|----------|
| **Data** | 标准化异构数据源（Slack、Jira、GitHub、Confluence） | Agent 找不到信息 | 统一 Ingestion Pipeline + 格式适配器 |
| **Retrieval** | 按需获取相关上下文 | 检索到无关或陈旧结果 | 混合检索 + 时效性权重 + 溯源 |
| **Reasoning** | 解决多源冲突（Slack 说 retry=3，Confluence 说 5） | Agent 自信地使用错误信息 | 冲突检测引擎 + 置信度评分 + 升级策略 |
| **Governance** | 每次查询执行权限管控 | Agent 泄露敏感数据 | ABAC + 查询级过滤 + 审计日志不可篡改 |

> **Google Research 2026**：上下文质量贡献了 **81%** 的可并行化 Agent 任务性能提升。

**面试中的应用**：当被问及 RAG 或 Agent 架构时，主动引入 Context Infrastructure Stack 的概念，说明你理解"检索到了 ≠ 上下文化了"。

### 3.0.2 AI-Native vs AI-Assisted——架构师的运营模型升级

| 维度 | AI-Assisted | AI-Native（2026 目标态） |
|------|-------------|------------------------|
| Agent 范围 | IDE 补全、聊天 | 全 SDLC 参与 |
| 输入 | 临时复制粘贴 | 标准化 Spec、Context Assembly |
| 输出 | 建议（接受/拒绝） | 带溯源和测试证据的 Artifact |
| 质量门禁 | 人工判断 | **Evals + CI 检查 + 人工审查** |
| 权限 | 用户自己的凭证 | **Scoped Token、沙箱环境** |
| 反馈环 | 无或非正式 | **Continuous Eval Regression Tracking** |

**AI SDLC 成熟度模型**：

| 阶段 | 治理方式 | 典型瓶颈 |
|------|----------|----------|
| **Stage 1: Adopt** | 无或可接受使用策略 | — |
| **Stage 2: Embed** | 自动化质量门禁 | Verification Tax（验证开销） |
| **Stage 3: Coordinate** | Policy-as-Code；跨框架合规 | 10% 生产力平台期 |
| **Stage 4: Orchestrate** | 基于身份的 Agent 治理；实时监控 | Governance Debt（治理债务） |

> **面试高分**：在"讲一个项目"时嵌入成熟度阶段的自我诊断，说明你不仅做了工程，还理解组织演进的节奏。

## 四、高难度专题（架构师二面/委员会拉分）

### 4.1 千万级 RAG + 多租户 SaaS（压轴系统设计）

**题目**（阿里/腾讯/豆包收集共识）：设计支持千万文档、高并发、低延迟、可审计的企业 RAG。

**推荐答法骨架**：

| 层 | 决策点（2026 升级） |
|----|---------------------|
| 接入 | 异步管道、格式适配器、病毒/PII 扫描 |
| 索引 | 按租户/业务线分片；HNSW/IVF 参数按规模调；冷热分离；**每租户独立 Collection + 共享 vLLM 推理** |
| 检索 | BM25+向量并行；RRF；Rerank 只对 Top-K；权限 pre-filter |
| 服务 | 无状态 API + 队列削峰；SSE 与同步 API 分流 |
| 治理 | Prompt/索引版本；golden 门禁；全链路 Trace；**Context Assembly 质量监控** |
| 演进 | v1 单体 Pipeline → v2 平台化 SPI → v3 联邦检索 → **v4 Context Infrastructure Stack** |

**资深追问**：共享向量集群 vs 每租户独立 Milvus——从 **隔离、成本、运维** 三维对比。

**2026 新增数据点**：
- 自托管 H100 裸金属 vs 托管 API：成本差 3x，延迟差 10x
- 推荐模式：**模型层共享（成本低）+ RAG 层每租户隔离（数据安全）**

---

### 4.2 大模型推理平台（GPU 调度）

**题目**（字节/谷歌）：多模型、动态扩缩、灰度、监控。

**核心机制（2026 升级）**：
- 请求入队 → Router（模型/能力/成本）→ **Continuous Batching** → GPU 池
- **控制平面与推理平面分离**：控制平面（Fargate 24/7）+ 推理平面（vLLM scale-to-zero，按秒计费）
- 延迟尖峰：**不阻塞全局**（队列隔离、优先级、小模型降级）
- 灰度：按租户/流量百分比切模型版本
- FinOps：GPU 利用率、$/token、空闲显存、**MFU**
- **投机解码**：小模型草稿 + 大模型验证，2-5x 推理加速
- **KV Cache 优化**：Prefix Caching + FP8 KV Cache + TurboQuant

**面经锚点**：`谷歌收集` GPU Scheduling / Batch Inference

### 4.3 AI Gateway + 信用/配额体系

**题目**（OpenAI 风格，`谷歌收集`）：高并发实时扣额度、按天刷新、防超卖。

**架构要点（2026 升级）**：
- 请求 **idempotency-key**
- 额度：Redis 原子扣减 + 异步对账 DB；热点租户分片
- 刷新：定时任务 + 版本化额度策略
- 与 Agent：**Tool 调用前预扣、失败后回滚**
- **2026 新增：Deterministic Gateway**——在边缘本地执行策略合规检查，不依赖概率性 RLHF
- **2026 新增：Token 级预算控制**——单次任务超过 N 个 Token 强制中断，每日按 Agent/工作流分配

---

### 4.4 路由与「第一步走错」的平台化治理

**现象**（腾讯&字节）：FAQ vs Tool 模糊 → 全链路跑偏。

**平台解法（2026 升级）**：
1. **三层 Router**：规则 → 轻量分类 → LLM Router（能规则不上 LLM）
2. **并行候选 + 业务状态仲裁**（未登录禁 Tool）
3. **低置信度澄清 / 兜底**
4. **Checkpoint 回滚 + 重规划**
5. **路由指标进 Dashboard**：误路由率、澄清率、转人工率
6. **（2026 新增）Deterministic Gateway 边缘决策**：将路由规则前移到网关，减少 LLM 调用的不确定性和成本

架构师要强调：**路由是平台能力，不是某 Bot 的 if-else**。

### 4.6 安全与合规架构（2026 监管升级）

| 主题 | 架构师答法（2026 升级） |
|------|----------------------|
| 五层安全 | 为何 RLHF 不够（腾讯 #7）：概率安全 ≠ 系统安全 |

### 4.8 Build vs Buy——2026 架构师的战略判断力

**2026 年"自研 vs 采购"已从二元选择演变为四层分级策略**：

| 层级 | 策略 | 适用场景 | 2026 动态 |
|------|------|----------|-----------|
| **Tier 1: Integrate & Activate** | 激活现有平台内置 AI（Copilot、Einstein、ServiceNow） | 非差异化功能 | 最快、最安全 |
| **Tier 2: Buy & Configure** | 采购行业专用工具 | 领域特定需求 | 注意集成债务 |
| **Tier 3: Build on Platform** | 在平台原语上构建（Azure OpenAI + 自定义工作流） | 核心差异化 | **2026 主流选择** |
| **Tier 4: Build from Scratch** | 完全自研 | AI 即产品本身 | 仅 AI-Native 公司 |

**7 因子评分模型**（1=强烈倾向采购，5=强烈倾向自研）：

| 标准 | 权重 | 关键问题 |
|------|------|----------|
| 战略差异化 | 20% | 是否创造竞争护城河？ |
| 数据独特性与复杂度 | 20% | 成功是否依赖专有、混乱、上下文丰富的数据？ |
| 集成深度 | 15% | AI 需与多少个系统可靠交互？ |
| 上市速度 | 15% | 价值实现以周还是季度衡量？ |
| 内部能力 | 10% | 团队有能力负责任地拥有此系统吗？ |
| 合规、安全与控制 | 10% | 需要什么治理边界？ |
| 长期经济性 | 10% | 预测规模下 24 个月哪个更便宜？ |

**判分**：<2.5=先采购 / 2.5-3.5=混合 / >3.5=自研值得认真考虑

> **面试高分**：在被问到"为什么选 X 不选 Y"时，使用此框架做结构化分析，而非仅仅技术对比。

## 五、资深推演题（架构师 / Staff 向，2026 大幅扩充）

> 建议每题 **5 分钟口述 + 1 页架构图**；强调 **权衡、演进、组织落地**。

### 5.1 平台与边界

1. **公司只允许一个 AI 中台，业务线如何接入又不被拖死？**  
   → 分层 SLA、自助配置上限、紧急 bypass 流程、季度架构评审

2. **抛弃 LangChain 自研平台，第一年 MVP 只做哪 3 件事？**  
   → Gateway+配额、Trace+评测、Tool Registry（MCP 优先）；编排可二期

3. **Python 智能层 + Java 治理层，边界怎么划？**  
   → Python：推理/RAG/Agent；Java：租户/权限/事务/工作流/HITL（腾讯&字节）

4. **100 租户共享 RAG，计费与隔离模型？**  
   → metadata + filter + 配额 + 审计；独立索引仅高合规租户

5. **Prompt 像代码一样管理？**  
   → Git + golden 回归 + 灰度 + 禁止线上手改；平台统一发布

6. **（新）Context Infrastructure Stack 如何落地？**  
   → 第一期：统一 Ingestion Pipeline（Data 层）；第二期：混合检索 + 时效性权重（Retrieval 层）；第三期：冲突检测引擎（Reasoning 层）；第四期：查询级 ABAC（Governance 层）

7. **（新）MCP 协议选型：自建 MCP Server 还是用现成的？**  
   → 内部工具（DB、API）自建 MCP Server；通用工具（搜索、代码执行）用社区生态；高风险工具额外加固 + mTLS

8. **（新）AI SDLC 成熟度从 Stage 2 到 Stage 3 的最大障碍？**  
   → 10% 生产力平台期：Eval 编写跟不上 AI 生成速度，Review 成为瓶颈。解法：Eval 生成也用 AI + 分层门禁（关键路径强制人工，非关键自动通过）

### 5.2 可靠性与容量

9. **SSE 断线防重复计费与重复 Tool？**  
   → idempotency-key + generation 状态机 + 重连只读（资深推演 #5 组织版）

10. **Agent P99 30s，平台如何定位？**  
    → 标准 Trace 字段；租户维度 P99 贡献；自动标记 Retry 风暴；**新增：Token 级成本归因定位昂贵 Agent**

11. **QPS ×100，最先爆的是哪？**  
    → 预演：向量检索扇出、LLM 队列、Redis 热 key、GPU 池；每类预案

12. **多 AZ / 推理容灾？**  
    → 模型多活、队列积压、降级到小模型、RAG 只读副本

13. **（新）GPU 利用率低（<30%）但 P99 延迟高，怎么调？**  
    → 分析 batch 长度分布 → 调整 max_batch_size / max_num_seqs → Continuous Batching 参数优化 → 考虑投机解码用小模型填充 batch

14. **（新）控制平面故障导致推理平面雪崩？**  
    → 控制平面与推理平面解耦：推理平面在控制平面故障时降级为静态路由 + 本地缓存策略，不自毁

### 5.3 安全、合规与治理

15. **监管要求「可解释溯源」，架构怎么满足？**  
    → chunk_id + 版本 + 检索分数 + Prompt 版本 + 模型版本 落审计库；**新增：Context Assembly 溯源链——展示每个上下文片段的来源和冲突解决过程**

16. **发现 Prompt 注入批量导出敏感数据？**  
    → 应用 DLP → Tool 收紧 → 租户冻结 → 模型（组织级 playbook）；**新增：MCP Server 审计日志回溯攻击路径**

17. **哪些能力必须做成「平台强制」而非业务可选？**  
    → 鉴权、审计、配额、HITL 钩子、评测门禁；**新增：Context Governance（查询级权限）、Agent 步数/Token 预算上限**

18. **（新）EU AI Act 2026.8.2 高风险义务生效，你的平台需要改什么？**  
    → 人类监督机制（HITL/HOTL 可配置）、停止与恢复能力、透明度日志、风险管理系统、合规报告 API

### 5.4 战略与组织（终面 / 国外大厂风格）

19. **团队只能投一个方向：自研 Runtime / 微调 / RAG 平台 / 评测，怎么排？**

20. **Harness 比模型更关键？** 结合你项目举一个反例（模型换代推翻假设）

21. **讲一个过度设计导致延误**（Anthropic 风格，`谷歌收集`）

    → 探索用轻量；生产用平台+Harness；避免全员 ReAct 裸奔

25. **（新）Build vs Buy：何时自建 AI 平台，何时买 SaaS？**  
    → 用 7 因子模型评分。**Tier 1 绝对不自己造（如 Office Copilot），Tier 3 值得投入（如核心业务流程 Agent）**

26. **（新）2026 DORA 报告：45% 高频 AI 用户每日部署，但 69% 报告 AI 生成代码至少一半有问题——你怎么设计治理架构？**

### 5.5 数据与 AI 融合架构

29. **向量库 + OLTP + 数据湖，职责边界？**  
    → OLTP 真相；向量检索副本；湖做训练/评测集；一致性策略各不同；**新增：Context Lake（非结构化知识湖）作为第四极**

30. **Agentic RAG / Graph RAG 何时进入平台标准能力？**  
    → 准入：golden 提升 > 成本阈值；否则业务自付费插件

31. **（新）实时 + 批量双通道数据摄入架构？**  
    → 实时通道（Kafka → Flink → 向量库增量）用于即时知识；批量通道（Spark → 数据湖 → 评测集）用于训练和 golden 更新

## 六、疑难杂症 → 架构师排查与机制速查（2026 扩展）

| 现象 | 平台第一反应 | 架构深挖 | 长期机制 |
|------|--------------|----------|----------|
| 全租户幻觉上升 | 查模型/Prompt 全局版本 | 是否灰度误放量；**Context Lake 是否被污染** | 发布门禁 + 金丝雀 + Context 对账 |
| 单租户召回差 | 租户索引/权限/filter | 知识未同步 | 租户 SLA + 对账任务 |
| P99 突增 | 全局 vs 单租户 Trace | GPU 队列/Retry 风暴；**投机解码草稿模型是否退化** | 队列隔离 + 步数预算 |
| 成本周报爆表 | $/tenant 排行 | Rerank/Planner/多 Agent；**有无 Model Tiering** | 配额硬顶 + 模型路由策略 + Token 预算 |
| JSON 格式崩 | 解析失败率 Dashboard | Schema/FC 覆盖率 | 平台强制 FC + 后置校验 |
| 路由大面积误杀 | 误路由率指标 | 规则未覆盖新意图 | Router 版本化 + A/B |
| 提示注入成功 | 攻击路径分类 | 文档/Tool/直连；**MCP Server 是否被横向移动** | 最小权限 + 红蓝演练 + mTLS |
| 索引更新仍答旧 | 版本号链路 | 缓存未失效；**Context Assembly 是否使用了旧版本** | 主动失效 + 增量管道 SLA + Context 版本绑定 |
| 多 Agent 踢皮球 | 轮次/仲裁日志 | 无 Manager 终审；**Plan-and-Execute 的 Re-planner 是否在兜圈** | 编排模板强制终止条件 + 步数硬上限 |
| GPU 利用率低 | batch 长度分布 | 调度策略；**推理平面 scale-to-zero 的冷启动是否过长** | Continuous Batching + 预热 + mmap 优化 |
| **（新）Agent 集体降智** | **Context 质量 Dashboard** | **Data 层是否有新数据源未标准化；Retrieval 是否召回过期上下文；Reasoning 层冲突率是否飙升** | **Context Infrastructure Stack 四层监控** |
| **（新）Eval 计费 > 推理计费** | **Eval Pipeline 成本归因** | **是否每次 PR 都跑全量 golden set；是否用了大模型做 Judge** | **分层 Eval：快速冒烟（小模型）+ 全量回归（定时）+ 大模型 Judge 仅可疑 case** |

## 八、各公司面试风格差异（2026 新增）

| 公司 | 面试侧重点 | 特别提醒 |
|------|-----------|----------|
| **字节跳动** | 极度追问工程细节，层层递进，考查是否真的在生产环境踩过坑 | 准备好每次事故的 **时间线 + 止血措施 + 长期机制** |

| **月之暗面/MiniMax** | 追问底层推理框架实现细节和模型能力边界 | vLLM/SGLang 源码级理解 |
| **Meta** | AI-Enabled Coding Round（60min）+ 传统系统设计 | 考察策略性 AI 使用，非"让 AI 替你做" |
| **Google** | 经典分布式 + GPU 调度 + 安全文化 | Scale Evidence 很重要 |
| **Anthropic** | 安全文化 + 过度设计反思 + HITL 设计 | 准备"你拒绝过什么需求"的真实案例 |
| **OpenAI** | 网关额度、高并发扣减、模型路由 | 防超卖、幂等、分布式一致性 |
