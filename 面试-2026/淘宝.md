## 面试问题与回答

### 1. harness工程你是怎么理解的
Harness是一站式CI/CD工程化交付平台，核心解决研发流程标准化、自动化、交付效率与环境管控问题。依托多年企业级工程化落地经验，它覆盖代码构建、多环境部署、灰度发布、资源治理、流程审批全链路，能统一中台、AI应用这类复杂项目的研发规范，减少人工运维成本，保障分布式服务与AI项目持续迭代的稳定性。

### 2. sdd开发有了解过吗，说一下
SDD是服务设计驱动开发，核心是前置化、标准化的业务设计模式。区别于先编码后补设计的开发方式，会先完成领域边界拆分、接口定义、数据模型设计、业务规则梳理，再开展编码开发。非常适配电商SaaS、分布式系统、AI多模块协作场景，能有效降低系统耦合、规避接口兼容问题，大幅提升复杂项目的可维护性与协作效率。

### 3. redis秒杀中Lua脚本做了哪些
秒杀场景下Lua脚本依靠Redis单线程特性实现全流程原子化执行，主要完成三件核心操作：
1.  原子校验商品剩余库存；
2.  单脚本内完成库存扣减操作；
3.  校验单用户限购规则、拦截重复下单请求。
彻底解决高并发秒杀下的超卖、库存扣减错乱、并发脏数据问题，兼顾高性能与数据准确性。

### 4. 异步创建订单是如何保证可靠性的
结合高并发电商落地经验，通过消息队列 + 事务兜底 + 幂等补偿全链路保障：
1.  采用事务消息，保证本地订单业务库操作与消息发送原子一致；
2.  消费端开启手动ACK，业务处理失败不提交确认；
3.  配置任务重试机制，临时异常自动重试；
4.  失败消息投递至死信队列，人工兜底排查；
5.  增加订单状态幂等校验，杜绝重复消费生成重复订单。

### 5. 商品多规格、商品与规格分表库存，如何保证扣减一致
依靠分布式事务 + 分布式锁 + 逆向回补 + 定时对账组合方案实现数据一致：
1.  核心交易场景使用Seata分布式事务，统一管控商品总库存、规格子库存两张表的更新；
2.  并发场景通过商品 + 规格维度分布式锁，防止并发扣减冲突；
3.  订单预扣库存，订单关闭/取消时自动逆向库存回补；
4.  定时任务定时校验双库存数据差异，自动修复数据偏差。

### 6. 定时任务关单未触发，怎么做兜底
搭建多重冗余兜底机制避免任务漏执行：
1.  主流程依赖分布式定时任务正常调度关单；
2.  下单同步发送延时消息MQ，到期自动触发超时关单，作为核心兜底；
3.  新增全量订单巡检补偿任务，定时轮询待支付超时订单；
4.  增加任务执行监控与告警，任务中断、执行超时时及时预警，支撑人工介入处理。

### 7. 单机定时任务有性能上限，有没有了解过并发的定时任务
单机定时任务受线程池、CPU、单机负载限制，存在任务串行、单点故障、无法横向扩容的短板。主流分布式并发定时任务框架如XXL-Job、Elastic-Job，核心能力包含任务分片并行、多机器负载均衡、故障转移、动态扩容、任务路由分发。可将关单、库存校验、数据统计等大批量任务拆分分片，多实例并行处理，彻底突破单机性能瓶颈。

### 8. 怎么保障数据一致性
根据业务场景分级选用方案：
1.  强一致性场景（订单、库存）：使用数据库本地事务、分布式事务保证实时一致；
2.  高并发最终一致场景：依靠异步消息、本地消息表、定时对账做异步补偿；
3.  并发控制：通过乐观锁、悲观锁、分布式锁防止并发脏写；
4.  常态化数据对账校验，定时修复数据偏差，平衡性能与数据安全。

### 9. caffeine的缓存怎么保证和redis一致性
Caffeine是本地内存二级缓存，结合Redis分布式缓存，通过统一更新策略保障一致：
1.  数据新增/修改/删除时，严格执行先更新数据库、再删除Caffeine本地缓存 + Redis缓存；
2.  所有缓存设置合理TTL过期时间，短期数据不一致自动消除；
3.  集群多节点通过消息广播，主动失效其他节点本地缓存；
4.  核心热点数据主动定时刷新，非热点数据依赖过期兜底。

### 10. 缓存穿透、击穿、雪崩
- **缓存穿透**：查询不存在的数据，请求直接打穿到数据库；解决方案：布隆过滤器拦截无效请求、空值缓存、接口参数合法性校验。
- **缓存击穿**：热点Key过期瞬间，大量并发请求直达DB；解决方案：分布式互斥锁、热点Key永不过期、定时主动刷新热点缓存。
- **缓存雪崩**：大量Key同时过期或Redis集群宕机；解决方案：过期时间随机打散、Redis集群高可用、服务熔断降级、多级缓存兜底。

### 11. 布隆过滤器原理
布隆过滤器是低内存消耗的概率型数据结构，底层由位数组 + 多个哈希函数构成。写入数据时，通过多个哈希算法计算不同下标，将位数组对应位置标记为1；查询数据时，校验所有对应下标是否全为1，全1则数据可能存在，存在0则一定不存在。优势是占用资源极低，缺点是存在少量误判率、不支持灵活删除，多用于RAG过滤无效检索、缓存穿透拦截。

### 12. 滑动窗口限流是如何实现的，和固定窗口比较有什么优势
滑动窗口限流将固定时间窗口拆分为多个细粒度时间片，随时间自动淘汰过期分片，基于Redis ZSet、时间片计数器统计实时请求量，实现流量平滑统计。相比固定窗口优势：
1.  彻底规避窗口临界时间流量突刺问题；
2.  流量统计更连续、精准；
3.  不会出现临界点流量瞬间翻倍压垮服务的情况，限流管控更贴合电商、AI接口真实流量特征。

### 13. 如果不想让限流有突刺，保证流量平稳，用什么
优先采用**漏桶算法 + 滑动窗口限流**组合方案。漏桶算法核心是匀速流出请求，强制削平流量波峰，保证流量平稳消化；搭配滑动窗口做单位时间总流量管控，既限制整体QPS上限，又杜绝瞬间流量突刺，非常适合AI接口调用、商品下单等高并发平稳限流场景。

### 14. RRF是为了解决什么问题的
RRF（倒数秩融合）是RAG多路召回的融合排序算法，核心解决向量检索与关键词检索打分体系不统一、结果排序失衡的问题。业务中通常会同时使用向量语义召回、BM25关键词召回，两类检索打分逻辑差异大，RRF无需复杂加权计算，依靠排名位次做融合，平衡语义匹配与关键词匹配效果，整体提升检索排序质量。

### 15. 基于交叉编码器的重排序，耗时是怎么样的
交叉编码器（Cross-Encoder）属于高精度精细重排模型，推理耗时相对偏高。对比向量粗排、RRF融合排序，单条文本对推理耗时提升数倍，单条耗时在毫秒至十毫秒区间，批量重排耗时会线性上涨。工程落地中，只会对Top50~Top100少量召回结果做交叉编码器重排，在精度和接口响应耗时之间做平衡，保障AI应用体验。

### 16. 向量数据库的索引结构说一下
向量数据库索引主要分为**精准索引**和**ANN近似近邻索引**两大类：
1.  精准索引：以FLAT全量遍历索引为主，100%召回率，适合小体量知识库；
2.  近似索引（海量数据主流）：包含HNSW、IVF_FLAT、IVF_PQ、Annoy。其中HNSW是AI RAG项目最常用索引，基于多层有向图结构，检索速度快、精度高；IVF系列依靠聚类分桶，适合超大规模海量向量存储，压缩内存开销。

### 17. pg的bm25检索和其他向量数据库的bm25有什么差异
1.  部署成本：PG原生BM25依托数据库插件实现，轻量化、无需额外部署；Milvus、ES等专用向量库BM25为独立优化组件；
2.  性能能力：中小数据量差距不大，高并发、亿级文档场景下，专用向量库分片分布式能力更强，检索吞吐更高；
3.  融合能力：PG仅支持纯文本全文检索，专用向量库原生支持BM25 + 向量混合检索，更适配RAG业务；
4.  运维扩展：PG一体化运维简单，向量库集群化能力更强，适合AI海量知识库长期迭代。

### 18. 召回率是怎么评估的，有测试集吗
会搭建业务专属评测测试集量化评估召回率。基于真实用户咨询问题、历史业务文档、商品知识库，人工标注标准问题 + 关联文档片段构建测试样本集。通过批量自动化检索任务，统计正确召回文档数量/应召回总文档数量计算召回率，同时覆盖常规问题、模糊语义、边界场景，保证评测结果真实有效。

### 19. 有什么核心评估指标
分为检索指标、大模型生成指标、工程指标、业务指标四大类：
1.  检索层：召回率、精确率、F1、NDCG、MRR；
2.  生成层：答案准确率、上下文相关性、幻觉率、回复完整度、推理耗时；
3.  工程层：接口QPS、响应延迟、并发稳定性、资源占用；
4.  业务层：问题解决率、重复提问率、人工转接率，兼顾技术落地与业务价值。

### 20. 意图识别是怎么做的
采用规则 + 小模型 + 大模型三层分层识别方案，兼顾性能与准确率：
1.  高频简单意图：通过关键词、正则、业务词典快速匹配，低耗时高稳定；
2.  通用标准意图：轻量分类小模型做统一分类；
3.  复杂模糊语义、隐含诉求：依靠LLM结合Prompt工程做语义理解；同时持续沉淀意图样本，迭代优化识别模型与规则。

### 21. 记忆机制是什么做的
AI应用记忆机制采用短期会话 + 长期持久化分层设计实现：
1.  短期会话记忆：基于Redis + 内存存储当前对话上下文，设置过期时间，保障交互响应速度；
2.  长期记忆：通过MySQL + 向量数据库持久化存储用户历史对话、业务偏好、个性化上下文；
3.  搭配上下文摘要、内容裁剪，避免Prompt超长，解决长对话模型推理过载问题。

### 22. 这个agent是ReAct的还是workflow的
项目中采用混合式Agent架构，根据业务场景差异化选型：固定标准化业务流程，比如商品库存查询、订单进度查询，使用Workflow固定编排保障稳定性；开放式复杂问答、多工具组合调用、多步骤复杂任务，采用ReAct架构，依靠大模型自主思考、决策与迭代执行，平衡稳定性与AI灵活性。

### 23. ReAct、Plan-Exec、workflow的agent有什么区别，适用于什么场景下
1.  **Workflow Agent**：固定节点编排、流程强约束、确定性高、幻觉少；适用于标准化高频固定业务，如单据查询、固定流程办理、规则化问答；
2.  **ReAct Agent**：思考-行动循环执行，自主决策工具与步骤；适用于开放式问答、不确定流程、多工具自由组合的轻量化复杂任务；
3.  **Plan-Exec Agent**：先全局规划完整步骤，再分步落地执行；适用于长链路复杂任务，如多文档分析、批量数据处理、复杂业务串联办理。

### 24. ReAct的agent能替代workflow和Plan-Exec吗
不能完全替代。ReAct高度依赖大模型自主决策，存在流程不可控、幻觉风险、执行不稳定的问题，不适合电商核心交易、强规则类业务；Workflow的强确定性、高性能、低风险是AI大模型无法替代的；Plan-Exec的全局规划能力，也优于原生ReAct。企业级AI落地最优方案是三者混合搭配，按业务复杂度灵活选用。

### 25. 如果召回率低怎么排查问题
按照RAG全链路逐层定位排查：
1.  数据层：文档切片长短不合理、文本脏数据、分词异常、知识库内容缺失；
2.  检索层：Embedding模型适配性差、BM25分词词典不全、单次召回条数配置过小；
3.  算法层：多路检索融合策略不合理、相似度阈值设置过高；
4.  工程层：向量索引未及时更新、缓存拦截、文档入库失败；
针对性优化切片、换Embedding模型、调整检索参数即可修复。

### 26. pdf解析遇到什么问题，怎么解决的
实战中遇到PDF解析核心问题：图文混排错乱、表格无法识别、扫描件纯图片无文本、水印乱码、排版碎片化。解决方案：
1.  结构化解析还原段落版式，修复排版错乱；
2.  复杂表格单独通过OCR + 表格解析组件提取；
3.  扫描件、图片类PDF接入OCR文字识别；
4.  统一清洗水印、特殊符号、乱码等脏数据；
5.  长文档预处理拆分，保证后续向量化与检索效果。

### 27. mcp怎么做的
MCP多上下文处理核心围绕AI长文本、多源信息融合管控设计。核心实现：
1.  文档预处理：文本清洗、智能切片、结构化规整；
2.  上下文筛选：通过RAG检索精准筛选关联上下文，过滤无效内容；
3.  上下文压缩：摘要精简、冗余内容裁剪，控制Prompt长度；
4.  多源融合：整合业务文档、对话历史、用户画像、工具返回数据；
5.  动态调度上下文，根据问题复杂度自适应加载，平衡效果与推理性性能。

### 28. 你觉得cli能替代mcp吗，有哪些优缺点
CLI完全无法替代MCP。
- CLI优点：轻量化接入、部署简单、开发成本低，适合单次极简AI调用场景；
- CLI缺点：无上下文管理、不支持多文档融合、长对话能力缺失、无法做内容裁剪与筛选；
MCP核心优势：具备多上下文调度、长对话记忆、多文档融合、上下文优化能力，是企业级RAG、智能体落地的核心能力，复杂AI业务必须依赖MCP支撑。

### 29. 你觉得ai生成的这个代码有问题的，存在哪些问题
AI生成代码普遍存在工程化短板：
1.  业务逻辑不严谨，缺少边界判断、异常捕获、参数校验；
2.  缺少工程规范，无日志埋点、无统一编码规范；
3.  高并发场景存在缺陷，缺少分布式锁、幂等、防重设计；
4.  性能较差，SQL低效、无缓存设计、资源冗余消耗；
5.  耦合度高，扩展性差，无法适配业务迭代，必须人工二次优化审核。

### 30. 项目中有些地方可以优化的
从AI能力、工程架构、业务落地三个维度优化：
1.  RAG层面：优化智能切片策略、替换高精度Embedding模型、升级多路召回融合算法；
2.  Agent层面：完善工具调用幂等、增加降级熔断、精简Prompt降低推理成本；
3.  工程层面：优化二级缓存策略、核心接口异步化、完善全链路监控告警；
4.  数据层面：实现知识库增量更新、向量索引定时刷新，提升整体稳定性。

### 31. 项目里你这个业务你觉得应该有那些表，哪些字段
以电商AI商品咨询RAG+Agent业务为例，核心数据表及关键字段：
1.  知识库文档表：id、文档标题、原文内容、切片文本、业务类型、更新时间、上下架状态；
2.  向量索引表：切片id、向量数据、Embedding模型标识、关联业务id、创建时间；
3.  对话会话表：会话id、用户id、对话角色、问答内容、上下文摘要、会话过期时间；
4.  工具调用记录表：会话id、工具名称、请求入参、响应结果、调用状态、异常信息；
5.  意图配置表：意图id、意图名称、关键词列表、绑定工具、识别优先级。

### 32. 项目里你这个业务流转流程是什么样的
整体为电商AI智能咨询全闭环流程：
1.  用户发起提问，网关完成限流、身份权限校验；
2.  意图识别模块解析用户诉求，区分普通问答、工具查询、复杂任务；
3.  RAG引擎执行向量 + BM25多路召回，经重排序筛选优质关联文档；
4.  Agent判断是否需要调用工具，完成库存、商品、订单等业务数据查询；
5.  大模型结合检索文档、工具结果、历史对话生成合规回复；
