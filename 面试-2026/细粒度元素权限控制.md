# 前端细粒度元素权限控制（工业级前后端完整方案）
## 一、核心概念与分级权限定义
细粒度权限：**路由 → 页面模块 → 按钮/图标/输入框/文本/区域** 全维度管控，支持隐藏、禁用、只读、可编辑、弹窗拦截等多种状态，区别于单纯路由鉴权。
权限主体：用户角色、数据权限、功能权限、操作权限四层隔离。

---
## 二、整体架构设计（前后端协同）
### 1. 架构分层
```
后端：权限数据源 + 鉴权校验 + 权限聚合接口
↓
网关层：统一拦截、Token校验、基础路由权限
↓
前端：权限解析 + 全局状态 + 指令/组件/路由 多层渲染控制
```

### 2. 权限数据模型（后端核心表设计）
#### （1）核心数据表（通用工业级设计）
1. **用户表 user**：用户ID、账号、关联角色ID
2. **角色表 role**：角色ID、角色名称（管理员/运营/普通用户等）
3. **权限菜单表 permission**（最小权限单元，**元素级粒度**）
| 字段 | 说明 |
| ---- | ---- |
| perm_id | 权限唯一标识（核心，前端绑定依据） |
| parent_id | 父级权限ID（树形结构，路由/模块/元素） |
| perm_type | 类型：1-路由 2-页面模块 3-按钮/元素 4-接口 |
| perm_code | 权限编码（字符串，前端硬编码匹配，如 `user:add`、`user:delete`） |
| perm_name | 权限名称 |
| status | 启用/禁用 |
| sort | 排序 |

4. **角色权限关联表 role_perm**：角色ID + perm_id（多对多）
5. **数据权限表 data_scope**：控制行数据可见范围（部门/个人/全部，和元素权限解耦）

#### （2）权限编码规范（统一约定，关键）
采用**`模块:页面:操作`** 三段式，全局唯一：
- 路由：`system:user:route`
- 页面模块：`system:user:search`
- 按钮：`system:user:add`、`system:user:edit`、`system:user:export`
- 输入框/控件：`system:user:input-phone`
- 区域：`system:user:card-info`

### 3. 后端接口设计（前端唯一数据源）
#### （1）登录后拉取权限（核心接口）
**接口**：`GET /api/auth/perm-info`
**入参**：Token（请求头）
**出参结构**（前端直接消费）：
```json
{
  "code": 200,
  "data": {
    // 1. 树形菜单（用于侧边栏、路由渲染）
    "routes": [...],
    // 2. 扁平化权限编码集合（前端核心校验源，数组/Set）
    "permissions": ["system:user:add", "system:user:edit", "system:user:query"],
    // 3. 角色列表
    "roles": ["admin"],
    // 4. 数据权限范围
    "dataScope": "ALL"
  }
}
```
> 设计原则：**后端只输出权限编码集合，不干预前端渲染逻辑**，解耦前后端。

#### （2）接口级鉴权（双重兜底）
所有业务接口后端增加**接口权限校验**：
- 前端即使绕过元素控制发起请求，后端直接返回 `403 无权限`，**防止前端权限篡改**（工业级必备兜底）。

---
## 三、前端整体方案（细粒度控制落地）
技术栈通用：Vue2/Vue3 + React 均适配，分为**四层拦截体系**，从粗到细逐级管控。

### 前置：权限状态管理
全局存储权限集合，推荐 `Set` 结构（查询 O(1)，性能最优）：
1. 登录成功后，调用权限接口，将 `permissions` 转为全局 `permissionSet`
2. 退出登录/Token过期，清空权限集合
3. 状态库：Vue → Pinia/Vuex；React → Redux/Zustand/Context

示例（Vue3 + Pinia）：
```ts
// stores/permission.ts
import { defineStore } from 'pinia'
export const usePermStore = defineStore('permission', {
  state: () => ({
    permissionSet: new Set<string>(),
    routes: []
  }),
  actions: {
    setPermissions(list: string[]) {
      this.permissionSet = new Set(list)
    },
    hasPerm(permCode: string): boolean {
      return this.permissionSet.has(permCode)
    }
  }
})
```

### 第一层：路由级别权限（页面访问控制）
目标：无权限用户**无法进入对应页面**，拦截路由跳转。
#### 实现方案：路由守卫 + 动态路由
1. 拆分路由：**静态路由**（登录、404、首页）+ **动态路由**（业务页面，由后端返回）
2. 流程：
   - 进入页面 → 路由前置守卫 → 检测Token
   - 无Token → 跳登录页
   - 有Token 但无权限数据 → 先请求 `/api/auth/perm-info` 获取权限 & 动态路由
   - 拼接完整路由，`addRoute` 注册动态路由
   - 校验当前路由是否在权限内，无权限跳 403/404

> 工业级注意：
> - 路由懒加载，分包优化首屏速度
> - 防止 `addRoute` 重复注册（路由去重）
> - 404 路由必须**最后注册**

---
### 第二层：页面模块/区域权限（整块DOM控制）
目标：控制**整个卡片、表单、弹窗、分区**显示/隐藏。
#### 方案1：通用权限组件（跨框架通用）
封装 `<Permission>` 高阶组件，接收 `perm-code` 属性，内部判断权限：
- 有权限：渲染默认插槽
- 无权限：渲染空 / 占位插槽

**Vue3 示例**
```vue
<!-- components/Permission.vue -->
<template>
  <slot v-if="hasPerm" />
  <slot name="fallback" v-else></slot>
</template>

<script setup lang="ts">
import { usePermStore } from '@/stores/permission'
const props = defineProps<{ permCode: string }>()
const permStore = usePermStore()
const hasPerm = permStore.hasPerm(props.permCode)
</script>
```

**使用**
```vue
<!-- 整个用户编辑区域无权限则隐藏 -->
<Permission perm-code="system:user:edit-area">
  <div class="edit-card">
    <el-form>...</el-form>
  </div>
</Permission>
```

**React 示例（函数组件）**
```tsx
// components/Permission.tsx
import { usePermStore } from '@/stores/permission'
interface Props {
  permCode: string
  children: React.ReactNode
  fallback?: React.ReactNode
}
export default function Permission({ permCode, children, fallback = null }: Props) {
  const { hasPerm } = usePermStore()
  return hasPerm(permCode) ? children : fallback
}
```

### 第三层：元素级细粒度控制（按钮、输入框、图标、文本）
这是**细粒度核心**，支持 4 种常用状态：`显示/隐藏`、`启用/禁用`、`只读`、`不可点击`。
提供两种工业级实现方式：**指令（推荐Vue）**、**组件封装（推荐React）**。

#### 方式1：Vue 自定义权限指令（最常用）
##### 1. 基础指令：控制 显示/隐藏
```ts
// directives/permission.ts
import { usePermStore } from '@/stores/permission'
export default {
  mounted(el: HTMLElement, binding: { value: string }) {
    const permStore = usePermStore()
    const permCode = binding.value
    if (!permStore.hasPerm(permCode)) {
      el.style.display = 'none'
    }
  }
}
```
全局注册后直接使用：
```vue
<!-- 新增按钮无权限隐藏 -->
<el-button v-perm="'system:user:add'">新增用户</el-button>
<!-- 删除按钮 -->
<el-button v-perm="'system:user:delete'">删除</el-button>
```

##### 2. 增强指令：支持 禁用/只读/隐藏 多模式
扩展指令参数，实现精细化状态：
```vue
<!-- 无权限则禁用按钮 -->
<el-button v-perm:disabled="'system:user:export'">导出</el-button>
<!-- 输入框无权限设为只读 -->
<el-input v-perm:readonly="'system:user:input-name'"></el-input>
```

#### 方式2：React 封装原子组件（统一接管UI元素）
对业务组件库（Ant Design/Arco）二次封装，内置权限判断：
```tsx
// components/PermButton.tsx
import { Button, ButtonProps } from 'antd'
import { usePermStore } from '@/stores/permission'

interface PermBtnProps extends ButtonProps {
  permCode: string
  mode?: 'hide' | 'disabled' // 无权限模式：隐藏 / 禁用
}

export default function PermButton({ permCode, mode = 'hide', ...rest }: PermBtnProps) {
  const { hasPerm } = usePermStore()
  const isAuth = hasPerm(permCode)

  if (!isAuth && mode === 'hide') return null
  return <Button {...rest} disabled={!isAuth || rest.disabled} />
}
```

业务使用：
```tsx
// 无权限直接隐藏
<PermButton permCode="system:user:add">新增</PermButton>
// 无权限置灰禁用
<PermButton permCode="system:user:edit" mode="disabled">编辑</PermButton>
```

#### 补充：纯文本/标签权限
直接使用 `v-if` / 三元表达式（简单场景）：
```vue
<span v-if="$perm.hasPerm('system:user:show-phone')">138****1234</span>
```

---
### 第四层：接口请求层兜底（安全防线）
前端元素控制只是**体验层**，必须结合接口拦截做安全兜底：
1. 前端 Axios/Request 封装：请求前可二次校验权限（可选）
2. **后端接口强制鉴权（必做）**：
   - 每个接口绑定权限编码
   - 用户发起请求，后端校验角色&权限，不通过返回 `403 Forbidden`
3. 前端统一捕获 403 响应：弹窗提示「无操作权限」

## 四、特殊场景解决方案（工业级高频需求）
### 1. 动态表格行内权限（行级元素）
场景：**不同行数据，按钮权限不同**（如自己的数据可编辑，他人数据仅查看）
方案：**元素权限 + 数据权限 组合判断**
```vue
<template>
  <el-table-column label="操作">
    <template #default="{ row }">
      <!-- 同时判断功能权限 + 该行数据权限 -->
      <el-button
        v-if="hasFuncPerm('system:user:edit') && hasDataPerm(row)"
      >编辑</el-button>
    </template>
  </el-table-column>
</template>
```

### 2. 弹窗/抽屉内部权限
弹窗属于动态渲染DOM，指令/组件依然生效，无需额外适配；
注意：弹窗打开时**不要重复请求权限**，复用全局权限集合。

### 3. 权限动态刷新（在线改权限，不登出生效）
业务场景：管理员后台实时修改角色权限，用户无需刷新页面立即生效。
方案：
1. 后端提供 `刷新权限` 接口
2. 前端监听（轮询 / WebSocket）权限变更事件
3. 重新调用权限接口，更新全局 `permissionSet`
4. Vue/React 响应式自动更新页面元素状态

### 4. 微前端场景下的权限隔离
微前端（qiankun/icestark）工业级方案：
1. **主应用**：统一登录、获取全局权限、下发权限数据到子应用
2. **子应用**：接收主应用权限，本地解析、渲染控制
3. 规则：主应用管控路由，子应用管控内部细粒度元素

### 5. 权限缓存与降级
- 本地缓存：登录后权限集合存入 `sessionStorage`，刷新页面优先读缓存，再异步校验后端有效性
- 降级策略：接口异常/超时，默认禁用所有操作按钮，保证系统安全

---
## 五、安全规范 & 避坑要点（工业级红线）
1. **禁止前端硬编码判断角色**
   ❌ 错误：`v-if="role === 'admin'"`
   ✅ 正确：统一使用**权限编码**判断，角色由后端映射为权限集合。

2. **永远不要信任前端权限**
   前端只做**UI展示拦截**，**所有操作接口后端必须二次鉴权**，防止篡改前端JS、伪造请求。

3. 权限编码全局唯一，文档统一维护
   前后端共用一份权限编码字典，避免编码不一致导致权限失效。

4. 禁用浏览器调试篡改DOM
   不要依赖 `display:none` 做安全控制，核心操作必须依赖后端接口鉴权。

5. 权限集合使用 `Set` 而非数组
   大数据量下查询性能差距极大，页面元素越多越明显。

6. 路由懒加载 + 权限分离
   避免一次性加载所有路由，提升首屏性能。

---
## 六、整体流程总结（完整链路）
1. 用户登录 → 校验账号密码 → 下发 Token
2. 前端路由守卫拦截 → 携带Token请求 `权限信息接口`
3. 后端根据用户角色，返回 **路由树 + 权限编码集合**
4. 前端注册动态路由，将权限集合存入全局状态
5. 页面渲染：路由→模块→元素 三层权限逐级控制显示/禁用
6. 用户执行操作（点击按钮）→ 前端校验权限 → 发起接口请求
7. 后端接口再次鉴权 → 通过则执行业务逻辑，不通过返回403
8. 前端捕获403，给出无权限提示

该方案支持从**页面、区域、按钮、输入框、行数据、接口**全链路细粒度管控，兼容单体应用、微前端、中后台复杂系统，是目前国内企业主流工业级实现。
