# 一、消息队列 + 异步任务（第一高频区）

## 1.1 必知方案对比

| 维度 | Redis MQ | RabbitMQ | RocketMQ | Kafka |
|------|----------|----------|----------|-------|
| 吞吐量 | 5万/s | 1万/s | 10万/s | 100万/s |
| 可靠性 | 低（断电丢消息） | 高 | 高（同步刷盘） | 高（ISR） |
| 事务消息 | ❌ | ❌ | ✅ 半消息+回查 | ❌（幂等替代） |
| 延迟消息 | ❌ | ✅ 死信插件 | ✅ 18个等级 | ❌ |
| 顺序消息 | ❌ | ✅ | ✅ 同一Queue | ✅ 同一Partition |
| 适用场景 | 轻量异步 | 业务强一致 | 交易、国内主流 | 大数据、日志 |

## 1.2 面试最常问（10题）

### Q1：消息不丢失怎么保证？（全链路三阶段）
```
生产端 → Broker → 消费端 全链路ACK

生产端：同步发送 + 确认回调 + 失败重试
Broker端：同步刷盘（RocketMQ flushDiskType=SYNC_FLUSH）
          + 多副本（主从同步，Kafka min.insync.replicas ≥ 2）
消费端：手动ACK（处理完再确认，不要自动ACK）
        + 消费失败写入死信队列，不丢消息
```

### Q2：消息重复消费怎么解决？（幂等）
```
本质：MQ保证 At-Least-Once，重复是必然的

方案（按可靠性排序）：
1. 数据库唯一索引：msg_id + biz_type 联合唯一键，插入即幂等
2. Redis SETNX：msg_id 做 key，消费前置 setnx 判断
3. 业务状态机：订单状态已支付 → 忽略重复支付回调
4. 版本号：消息带 version，DB update ... where version < #{msg.version}
```

### Q3：消息积压怎么处理？（线上事故第一优先级）
```
1. 紧急止血：临时增加消费者实例（扩Pod × 10），快速消化积压
2. 降级：非核心业务消息暂停消费，优先核心链路
3. 转移堆积：积压消息转存另一个Topic，原Topic正常消费，异步补消费转存消息
4. 长期方案：
   - K8s HPA/KEDA 基于消息积压自动扩缩
   - 告警阈值：积压 > 10w 钉钉告警
   - 死信队列兜底
```

### Q4：RocketMQ 事务消息原理？（手画流程图）
```
半消息机制（两阶段提交）：

    Producer → Broker（Half消息，消费者不可见）
         ↓
    执行本地事务
    ├── 成功 → COMMIT → Broker标记可消费 → Consumer可见
    ├── 失败 → ROLLBACK → Broker删除
    └── 未知 → Broker定时回查（checkLocalTransaction）→ 决定COMMIT/ROLLBACK

面试加分：回查至少15次，超时未响应直接标记为ROLLBACK
```

### Q5：Kafka 为什么吞吐这么高？
```
1. 顺序写磁盘：append only，顺序写性能 ≈ 内存随机写
2. 零拷贝：sendfile 系统调用，数据从PageCache → 网卡，不经过用户态
3. PageCache：依赖OS页缓存，读多写少场景命中率高
4. 批量+压缩：Producer批量发送 + Snappy/GZIP压缩
5. 分区并行：多Partition并发读写
```

### Q6：异步任务框架选型对比

| 平台 | 语言支持 | 定时 | 编排 | 重试 | 监控面板 | 适用规模 |
|------|---------|------|------|------|---------|---------|
| Celery | Python | ✅ | ❌ | ✅ | Flower | 中小 |
| XXL-JOB | Java为主 | ✅ | ❌ | ✅ | ✅ | 中小-中 |
| PowerJob | 任意语言 | ✅ | ✅DAG | ✅ | ✅ | 中大 |
| Temporal | Java/Go/Python/JS/.NET | ✅ | ✅Workflow | ✅ | ✅ | 超大 |

### Q7：定时任务多实例重复执行怎么解决？
```
单机 @Scheduled/Crontab 问题：多实例部署，每个实例都执行

方案（按推荐度排序）：
1. XXL-JOB/PowerJob：调度中心只分发给一个Worker执行
2. Redis分布式锁：任务执行前 SETNX 获取锁，执行完释放
3. 数据库乐观锁：UPDATE task SET status=1 WHERE status=0 AND id=#{id}
4. 分片广播：将任务数据按分片分配到不同实例，各执行一部分
```

### Q8：中大厂统一任务体系怎么设计？
```
统一消息总线（一套RocketMQ集群）
    +
统一调度平台（Temporal/PowerJob，管所有语言定时+编排）
    +
各语言Worker（仅负责消费执行业务逻辑）
    +
统一可观测（OpenTelemetry TraceID全链路透传）
    +
Protobuf统一序列化（跨语言强类型消费）

消除：Celery/Asynq/XXL-JOB 各玩各的割裂
```

### Q9：延迟任务怎么实现？
```
方案（按精度排序）：
1. RocketMQ 延迟等级（18个等级：1s 5s 10s 30s 1m ... 2h）
2. Redis ZSET：score=执行时间戳，定时轮询弹出到期任务
3. RabbitMQ 死信队列 + TTL
4. 时间轮算法（Netty HashedWheelTimer，内存高效）
5. 数据库轮询：SELECT ... WHERE execute_time <= NOW()，简单但性能差
```

### Q10：消息顺序性怎么保证？
```
场景：同一个订单的创建→支付→发货必须顺序消费

方案：
RocketMQ：同一业务Key（订单ID）路由到同一个MessageQueue
          + Consumer端串行消费（单线程消费该Queue）

Kafka：同一Key路由到同一Partition
        + 单Partition内有序

注意：顺序消费会牺牲并行度，非必要不开启
```

# 二、分布式基础理论（面试必考，答错一票否决）

## 2.1 CAP 定理

```
C（Consistency）一致性：所有节点同一时刻数据一致
A（Availability）可用性：服务一直可用，正常响应
P（Partition Tolerance）分区容错：网络分区时系统仍能工作

核心结论：P 必须选（网络故障不可避免），实际是 CP vs AP 的权衡

CP：Zookeeper（主挂掉选举期间不可用）、etcd
AP：Eureka（优先保证可用，数据可能不一致）、Redis Cluster
```

## 2.2 BASE 理论

```
Basically Available（基本可用）：系统故障时允许降级
Soft State（软状态）：允许系统存在中间状态
Eventually Consistent（最终一致性）：不要求实时一致，但最终一致

是 CAP 中 AP 的延伸，指导分布式事务和MQ的最终一致性方案
```

## 2.3 Raft 一致性算法（高频手写/口述）

```
角色：Leader / Follower / Candidate

选举过程：
1. Follower 超时未收到 Leader 心跳 → 变为 Candidate，Term+1
2. Candidate 向其他节点索要投票
3. 收到多数票 → 成为 Leader，开始发送心跳
4. 选举期间出现平票 → Term+1 重新选举

日志复制：
1. Client 写请求 → Leader
2. Leader 追加日志到本地，并行发送 AppendEntries 给 Followers
3. 多数 Follower 确认后 → Leader 提交 → 通知 Follower 提交
4. 提交后才返回 Client 成功

面试必答：Raft 保证的是 顺序一致性（不是线性一致性）
```

## 2.4 分布式ID生成

| 方案 | QPS | 趋势递增 | 缺点 |
|------|-----|---------|------|
| UUID | 无限 | ❌ | 非单调，B+Tree插入性能差 |
| 数据库自增 | 低 | ✅ | 单点瓶颈 |
| Redis自增 | 中 | ✅ | 持久化依赖RDB |
| 雪花算法 | 高（单机400w/s） | ✅ | 时钟回拨问题 |
| 号段模式(Leaf) | 高 | ✅ | 需DB存储号段 |

雪花算法结构：1bit + 41bit时间戳 + 10bit机器ID + 12bit序列号
时钟回拨解决：①短期等待 ②备份节点替换 ③扩展位借用

# 三、微服务治理（分布式服务核心）

## 3.1 RPC vs HTTP

| 维度 | RPC（Dubbo/gRPC） | HTTP REST |
|------|-------------------|-----------|
| 协议 | TCP长连接 + 二进制序列化 | HTTP短连接 + JSON |
| 性能 | 高（Protobuf + 连接池） | 低（文本序列化） |
| 服务治理 | 内置（注册发现、负载均衡） | 需网关层补齐 |
| 跨语言 | gRPC支持 | 天然支持 |
| 适用 | 内部微服务通信 | 对外API、浏览器 |

## 3.2 服务注册发现（必问）

```
角色：
- Provider：启动时注册到注册中心，定时心跳续约
- Consumer：从注册中心订阅服务列表，本地缓存 + 变更通知
- Registry：Nacos / Eureka / Consul / etcd

心跳机制：
- 临时节点（ephemeral）：断连自动删除
- 健康检查：TCP/HTTP 主动探测

CP vs AP 选型：
- Nacos：可切换 CP（RAFT） / AP（Distro）
- Eureka：纯 AP（自我保护机制，宁可保留旧节点也不剔除）
- ZK/etcd：CP（主挂掉选举期间不可用）
```

## 3.3 熔断 & 限流 & 降级（防雪崩三件套）

### 熔断（断路器模式）
```
三状态：CLOSED → OPEN → HALF_OPEN → CLOSED
- CLOSED：正常请求，记录失败次数
- OPEN：失败率达阈值 → 直接拒绝请求，快速失败
- HALF_OPEN：冷却时间到 → 放少量请求试探
  - 成功 → CLOSED
  - 失败 → 回到 OPEN

框架：Sentinel（阿里，推荐）、Hystrix（停更）、Resilience4j
```

### 限流算法
```
令牌桶：匀速生成令牌，请求取令牌，取不到限流
        允许突发流量（桶容量 > 生成速率）

漏桶：请求进桶，匀速流出
      流量绝对平滑，不允许突发

滑动窗口：统计过去N秒请求数，超过阈值限流
          精度比固定窗口高

Sentinel 实现：滑动窗口 + 令牌桶结合，支持单机/集群限流
```

### 降级策略
```
1. 返回兜底值：库存查询失败 → 返回缓存库存
2. 返回默认值：用户信息失败 → 返回匿名用户
3. 功能降级：推荐服务挂了 → 返回热门商品
```

## 3.4 全链路追踪

```
TraceID（全局唯一）→ SpanID（单次RPC调用）→ ParentSpanID（上级调用）

一次请求：TraceID 串起所有 Span，形成调用链 DAG

工具：SkyWalking（Java首选，agent无侵入） / Jaeger（Go首选） / Zipkin

面试重点：TraceID 如何在跨服务/跨MQ/跨线程间透传？
- HTTP：Header 透传（X-Trace-Id）
- RPC：metadata/attachment 透传
- MQ：消息头透传
- 线程池：MDC.put / Context propagate
```

## 3.5 分布式事务（超高频，必须有实战方案）

### 最终一致性方案（生产首选）

**方案一：RocketMQ 事务消息**
```
适用：跨服务异步调用，比如下单 → 发短信/发券/加积分

流程：
OrderService → 发半消息到MQ → 执行本地事务（创建订单）
  → 成功：COMMIT（下游Consumer消费发短信）
  → 失败：ROLLBACK

下游Consumer做好幂等 + 消费失败重试
```

**方案二：本地消息表 + 定时补发**
```
适用：没有RocketMQ，只有RabbitMQ/Kafka

流程：
1. 本地事务同时写业务表 + 消息表（同库同事务）
2. 定时任务扫描消息表 status=pending，发送MQ
3. 发送成功 → 标记已发送
4. 发送失败 → 重试N次 → 人工介入

优点：不依赖MQ事务能力，通用
缺点：侵入业务代码，需要定时任务
```

### 强一致性方案（理解原理，面试讲清楚）

| 方案 | 原理 | 优点 | 缺点 | 适用 |
|------|------|------|------|------|
| Seata AT | 自动生成回滚SQL | 无侵入 | 性能损耗 | 单体拆微服务过渡 |
| Seata TCC | Try/Confirm/Cancel | 性能好 | 侵入大 | 资金交易 |
| Seata Saga | 状态机 + 补偿 | 长事务 | 无隔离性 | 长流程编排 |
| XA | 两阶段提交 | 强一致 | 资源锁定长 | 传统企业 |

### 对比：事务消息 vs TCC vs Saga

| 维度 | 事务消息 | TCC | Saga |
|------|---------|-----|------|
| 一致性 | 最终 | 最终 | 最终 |
| 隔离性 | 无 | 有（预留资源） | 无 |
| 侵入性 | 低（只改生产者） | 高（3个接口） | 中（补偿接口） |
| 适用场景 | 异步解耦 | 资金扣减 | 长流程 |
| 回滚方式 | 不发消息 | Cancel | 补偿 |

# 四、分布式锁（高频考判断方案优劣）

## 4.1 三种实现对比

| 维度 | Redis（Redisson） | Zookeeper | 数据库 |
|------|-------------------|-----------|--------|
| 原理 | SET NX + Lua | 临时顺序节点 + Watch | SELECT FOR UPDATE |
| 性能 | 高 | 中 | 低 |
| 自动释放 | 看门狗续期 | 临时节点断连删除 | ❌ 需手动 |
| 可靠性 | AP（主从切换可能丢锁） | CP | CP |
| 公平锁 | ✅ 红锁 | ✅ 顺序节点 | ❌ |
| 适用 | 高并发、短锁 | 强一致、长锁 | 简单场景 |

## 4.2 Redis 分布式锁演进

```
V1（错误）：SETNX + DEL
  问题：锁超时任务没执行完，锁被误删 → 并发安全问题

V2：SET key value NX EX 30 + Lua脚本原子释放
  问题：锁过期时间不好设置

V3（Redisson）：看门狗自动续期
  原理：加锁时启动WatchDog线程，每10s续期
       释放锁时停WatchDog
       业务执行多久，锁就续多久

V4（Redlock）：多Redis实例，向N个独立Redis获取锁
  过半获取成功 → 加锁成功
  解决：Redis主从切换丢锁问题
  争议：Martin Kleppmann认为Redlock不安全
```

# 五、缓存高并发（Redis 面试重灾区）

## 5.1 缓存三问题

```
穿透：查不存在的数据 → 请求直达DB
  方案：①布隆过滤器 ②缓存空值（短TTL）

击穿：热点Key过期 → 大量请求直达DB
  方案：①互斥锁（单线程查DB回写缓存） ②永不过期+异步更新

雪崩：大量Key同时过期 → DB瞬间压垮
  方案：①过期时间加随机值 ②多级缓存 ③限流熔断
```

## 5.2 缓存更新策略

```
Cache Aside（最常用）：
  读：查缓存 → 未命中 → 查DB → 写缓存
  写：更新DB → 删除缓存（不是更新缓存）
  为什么删而不是更新？避免了并发写导致的缓存与DB数据不一致

延迟双删：
  删缓存 → 更新DB → 睡几百ms → 再删缓存
  解决：更新DB期间有请求读到旧数据写回缓存

Read/Write Through：缓存层直接对接DB，业务只读缓存
Write Back：只写缓存，异步刷盘（有丢数据风险）
```

## 5.3 Redis 高级数据结构（加分项）

| 结构 | 用途 | 场景 |
|------|------|------|
| BitMap | 位图 | 签到统计、在线状态、布隆过滤器 |
| HyperLogLog | 基数统计 | UV统计（0.81%误差） |
| GEO | 地理位置 | LBS附近的人、门店 |
| Stream | 消息队列 | 轻量MQ（ConsumerGroup、ACK） |
| Sorted Set | 有序集合 | 排行榜、延迟队列 |

## 5.4 大Key & 热Key

```
大Key（单个Key的Value过大）：
  危害：网络阻塞、主从同步延迟、rehash卡顿
  排查：redis-cli --bigkeys / memory usage
  方案：①拆分（user:1 → user:1:basic + user:1:profile） ②压缩

热Key（单Key QPS过高）：
  危害：单分片CPU打满
  方案：①多级缓存（本地Caffeine + Redis） ②读写分离 ③key拆分多副本
```

## 6.1 分库分表必问

```
分库分表时机：单表 > 500万行 或 > 2GB → 考虑分表
           单库 QPS > 2000 → 考虑分库

垂直分库：按业务拆（订单库、用户库、商品库）
水平分表：按分片键拆（订单表_0、订单表_1...）

分片算法：
- 取模：id % N，扩容需迁移数据
- 哈希取模：hash(id) % N，分布均匀
- 范围：按时间/ID范围，扩容方便但热点
- 一致性哈希：扩容迁移数据量小

中间件：ShardingSphere-JDBC / ShardingSphere-Proxy / MyCat
```

### 分库分表后的问题（必考）
```
1. 分布式ID：不能用自增主键 → 雪花算法 / Leaf号段
2. 跨分片查询：不能随意JOIN/分页
   - 关联查询：冗余字段 / 全局表 / 绑定表
   - 分页：ShardingSphere自动归并排序，深分页性能差
3. 跨分片事务：不推荐，改用最终一致性（MQ）
4. 扩容：一致性哈希 + 双写迁移
```

## 6.2 MySQL 核心

```
索引：B+Tree原理、聚簇索引 vs 非聚簇索引、覆盖索引、最左前缀
事务：ACID、MVCC（ReadView + Undo Log）、四种隔离级别
锁：行锁（Record/Gap/Next-Key）、表锁、死锁排查（SHOW ENGINE INNODB STATUS）
慢SQL：EXPLAIN（type/rows/Extra）、索引失效场景、大表DDL
```

# 七、稳定性 & 高可用

## 7.1 三大经典问题排查

```
缓存：穿透/击穿/雪崩/大Key/热Key （方案见上）

数据库：慢SQL（EXPLAIN type=ALL）→ 加索引
         锁等待（SELECT FOR UPDATE 范围过大）→ 减小锁定范围
         主从延迟（大事务/大表DDL）→ 拆分事务
         热点行（秒杀单行更新）→ 排队+Redis预扣

MQ：消息堆积 → 扩容+降级
     重复消费 → 幂等
     消息丢失 → 全链路ACK
     顺序错乱 → 单Partition+业务状态机
```

## 7.2 容灾架构

```
同城双活：两个机房距离 < 50km，数据同步延迟 < 2ms
异地多活：跨城市，数据冲突是大难题（需要全局ID+向量时钟/CRDT）

流量切换：DNS切换 / 流量调度平台一键切流
故障演练：定期混沌工程（ChaosBlade）
```

## 7.3 监控告警

```
黄金指标（RED/USE）：
- Rate（QPS）、Error（错误率）、Duration（延迟P99）
- Utilization（CPU/Mem）、Saturation（连接池/MQ堆积）、Errors

技术栈：Prometheus + Grafana（指标） + ELK（日志） + 钉钉/企业微信告警
告警分级：P0（线上故障）、P1（严重）、P2（警告）
告警收敛：同一故障5分钟内合并，防止告警风暴
```

# 八、面试场景题（高频综合题）

### 场景1：设计一个秒杀系统
```
核心思路：削峰 + 限流 + 预热

1. 前端：按钮置灰 + CDN静态化 + 验证码防刷
2. 网关：限流（令牌桶 1000 QPS） + 黑名单
3. 业务：Redis预扣库存（Lua原子操作）→ MQ异步下单
4. DB：最终减库存，防超卖（UPDATE ... WHERE stock >= #{num}）
5. 分布式锁：Redisson锁商品ID
6. 缓存预热：活动开始前加载库存到Redis
7. 削峰：验证码 + 排队（Redis List/Stream）
```

### 场景2：下游服务挂了，你怎么保证不拖垮上游？
```
1. 熔断（Circuit Breaker）：连续失败N次 → 直接快速失败
2. 降级：返回兜底数据 / 默认值
3. 限流：超过阈值拒绝请求
4. 隔离：线程池隔离（不同下游用独立线程池）
5. 超时控制：连接超时 + 请求超时，不无限等待
```

### 场景3：设计一个分布式任务调度平台
```
要素：
1. 调度中心：多副本HA（选主），时间轮 + DB持久化任务定义
2. Worker：注册发现 + 心跳 + 负载均衡
3. 任务分片：大任务拆N份，分发到不同Worker
4. 失败重试：指数退避（1s/5s/15s/30s...），最多N次
5. 监控：任务执行耗时、成功率、积压量
6. 统一化：支持Java/Go/Python Worker，Protobuf定义任务接口

可以参考：XXL-JOB（中小） / PowerJob（中大） / Temporal（超大）
```

### 场景4：线上突然大量超时，你怎么排查？
```
1. 看监控：Grafana看QPS/延迟/错误率，定位是哪个服务/接口
2. 看机器：CPU/Mem/GC频率/连接池，确认是否资源瓶颈
3. 看链路：SkyWalking/Jaeger找最慢的Span，定位是哪个下游或DB
4. 看日志：ELK搜索ERROR，找具体异常堆栈
5. 看DB：慢查询日志 + 锁等待 + 连接数
6. 看MQ：积压量，消费速率
7. 止血优先：限流、熔断、降级 → 然后定位根因
```
