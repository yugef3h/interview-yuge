# 面试题库梳理：高频 · 高难度 · 资深推演

> 基于 `面试-2026/` 目录下 17 份文档（腾讯/字节/快手/阿里/智谱/国外大厂面经 + 豆包体系化总结 + 实习感悟）交叉归纳。  
> 目标读者：**2 年+ AI 应用/全栈、冲资深/40K 档**，需要应对大厂「连环追问 + 根因深挖 + 线上疑难杂症」。

---

## 一、怎么用这份文档

| 层级 | 含义 | 准备方式 |
|------|------|----------|
| **高频** | 多场面经重复出现，一面必问 | 每个主题准备「30 秒概念 + 3 分钟项目案例 + 1 个数据」 |
| **高难度** | 二/三面拉分，答浅立刻追问 | 按「现象 → 根因 → 体系方案 → 线上验证」四段式 |
| **资深推演** | 文档未直接出现、但大厂资深岗合理会问 | 结合你的项目做「如果流量 ×100 / 如果换模型 / 如果合规审计」预演 |

**大厂深度思考的标准**：不是背定义，而是能讲清 **为什么这样设计、放弃了什么、线上怎么验证、失败了怎么兜底**。

## 二、高频考点（按出现频次排序）

### 2.1 概念辨析（几乎每场必问）

| 问题 | 出现来源 | 高分要点（一句话） |
|------|----------|-------------------|
| LLM / RAG / Agent 三者关系 | 豆包总结、豆包整理、腾讯&字节、智谱 | 组件 vs 系统：LLM=推理核，RAG=记忆增强，Agent=感知·规划·工具·反思闭环 |
| Agent = LLM + 工具调用？ | 豆包总结、常见问题 | **否**。缺规划、状态记忆、反思纠错，只是插件调用 |
| Agent vs Workflow | 腾讯&字节、字节1/2面、淘宝 | 控制权：代码预定义 vs LLM 动态决策；生产多为**混合** |
| ReAct / Plan-Execute / Workflow 选型 | 淘宝、豆包整理、字节2面 | 固定流程→Workflow；开放探索→ReAct；长链路→Plan-Execute；**不能互相替代** |
| LangChain vs LangGraph | 腾讯&字节、字节2面、飞书2面 | Chain=线性能力库；Graph=状态机+DAG+Checkpoint+失败恢复 |

**连环追问模板**：
1. 你项目里哪些节点 Workflow，哪些交给 LLM？
2. 第一步路由错了，后面全偏，怎么回退？
3. 不用框架，手写 Agent 最少要实现什么？

---

### 2.2 RAG 全链路（最高频技术栈，5~8 轮追问）

**标准流程（需能默写）**  
文档解析 → 清洗去重 → 分块 → Embedding → 向量库+元数据 → Query 改写 → 稠密+稀疏混合召回 → RRF/融合 → Rerank → 上下文组装 → LLM 生成 → 引文溯源 → 合规校验

| 子题 | 高频细节 | 典型追问 |
|------|----------|----------|
| **分块** | 固定/语义/层级；overlap 10%~20%；代码表格单独切 | 双栏 PDF、跨页表格怎么处理？ |
| **Embedding** | BGE/商用；中文场景；领域术语 | 换模型后索引要不要重建？ |
| **向量库** | FAISS/Milvus/Chroma/PGVector；HNSW/IVF | 千万文档、高并发怎么选型？ |
| **混合检索** | BM25 + 向量；RRF 解决打分体系不统一 | BM25 已经很好，还要向量吗？ |
| **Rerank** | Cross-Encoder；只对 Top50~100；毫秒~十毫秒/条 | 如何平衡重排质量与 TTFT？ |
| **Metadata** | doc_id、权限、版本、章节、过期时间 | 多租户检索怎么做权限过滤？ |
| **评测** | Recall@K、MRR、NDCG；RAGAS（忠实度、相关性） | golden set 怎么建？Bad Case 怎么回流？ |
| **缓存** | Embedding/Prompt Cache | 文档更新后如何保证缓存一致性？命中率怎么优化？ |

**各面经原题锚点**：字节1面（RRF、向量索引）、淘宝（RRF/HNSW/召回率评估）、腾讯2面（召回率低、忠诚度低、TTFT）、飞书2面（合同 RAG 全链路）

### 2.3 Agent 工程（2025–2026 核心）

| 主题 | 高频问法 | 要点 |
|------|----------|------|
| **三层记忆** | 短/长/工作记忆分别存什么？ | 短期=对话上下文；长期=向量库+RAG；工作=任务状态快照（断点续跑） |
| **记忆压缩** | 工具返回占满窗口怎么办？ | 摘要、滑动窗口、结构化裁剪、按 relevance 丢弃 |
| **短→长转化** | 什么条件下晋升长期记忆？ | 用户偏好、重复事实、任务结论；需 TTL+去重+重要性打分 |
| **Tool Calling** | Schema、失败重试、危险工具 | Schema+函数调用+服务端分发；白名单+参数校验 |
| **MCP vs Function Call vs CLI** | 快手1面、淘宝、字节1面 | MCP=标准化工具/上下文协议；CLI 无多轮上下文管理 |
| **多 Agent** | 几种角色、怎么调度、怎么终止 | Manager-Worker 为主；结构化消息；中控编排、禁止横向乱通信 |
| **稳定性** | 死循环、步骤漂移、模型不听指令 | 状态机+重复检测+步数上限+目标锚定+Checkpoint |
| **安全** | 提示注入、HITL | 五层：Prompt 护栏、工具白名单、审计、分级审批、上下文隔离 |
| **评测** | 怎么判断 Agent 好用 | golden set + 通过率 + 线上 badcase 回流 + A/B |

**各面经原题锚点**：腾讯元宝1/2面（记忆压缩、OpenClaw/Claude Code）、快手1/2面（Skill、沙箱、Harness）、字节2面（Tool 设计、模型不听指令）

### 2.4 框架与产品认知

| 问题 | 要点 |
|------|------|
| OpenClaw 架构 / 为什么 ReAct 就够 | 轻量 Agent Loop；对比 Claude Code 的分层记忆与复杂编排 |
| Claude Code Memory / Prompt 设计 | 项目规则 vs 用户偏好 vs 会话隔离；多轮任务不丢 |
| Harness Engineering | Agent 外层工程约束：CI/CD、评测、灰度、Trace，比模型更决定稳定性 |
| 扣子 vs LangGraph | 低代码快但难深度定制；生产复杂流用代码框架 |
| Prompt 边界 | 不能替代状态机、权限、业务校验；越长噪声越大 |

### 2.5 后端 + 八股（AI 岗仍高频）

> 实习感悟：**AI 应用开发本质是后端工程师**；阿里/字节/快手/腾讯二面均考。

| 类别 | 高频题 |
|------|--------|
| **Redis** | 数据结构选型；会话/Checkpoint；限流；Stream vs Kafka；穿透/击穿/雪崩；Caffeine 二级缓存一致性 |
| **网络** | TCP 三次握手/四次挥手/TIME_WAIT；HTTPS 握手 |
| **OS** | 进程/线程/协程；上下文切换；虚拟内存；死锁 |
| **MySQL** | B+ 树索引；存储引擎 |
| **Python** | GIL |
| **Transformer** | Self-Attention；MHA；GPT vs BERT；SFT/RLHF/DPO |
| **算法** | LRU、反转链表、合并有序链表、最长回文子串、单例 |

**与 AI 的结合问法**（淘宝/腾讯&字节）：Redis 在 Agent 里管什么？滑动窗口限流为什么比固定窗口好？

**必须准备的量化指标**：Recall@K 提升、幻觉率下降、TTFT、人工转接率、Token 成本、QPS/延迟。

## 三、高难度专题（二/三面拉分 + 线上真实痛点）

### 3.1 长任务 Agent：漂移 · 死循环 · 断点续跑

**现象**：步骤偏离目标、重复调同一工具、上下文被中间结果撑爆。

**根因**：
- 初始目标被挤出上下文窗口
- 单步 ReAct 缺乏全局约束
- 无执行轨迹监控

**体系解法**（多文档共识）：
1. 每轮 Prompt 锚定原始目标 + 里程碑
2. 上下文摘要压缩（保留目标/决策/异常）
3. 状态机记录 tool 调用序列 → 重复调用检测
4. 硬边界：max_steps / max_retry / timeout
5. Checkpointer 快照 → 中断恢复
6. 低置信度 → 澄清节点 / 人工接管

**资深追问**：反思结果写回上下文会不会污染下一轮？—— 要结构化存储、与对话历史隔离。

---

### 3.2 百级工具调度：准确率暴跌

**解法链**（字节/豆包/腾讯共识）：
1. 工具分层分类（先选域再选工具）
2. 工具描述向量化 → Top-N 预筛选
3. JSON Schema 强校验 + 少样本示例
4. 异常分类：参数错 / 超时 / 业务失败 → 不同重试策略
5. 服务端分发，不直接把全量工具暴露给模型

---

### 3.3 路由错误导致全链路跑偏

**腾讯&字节原题**：FAQ vs Tool 边界模糊怎么办？

**解法**：
- 意图打分 + 并行候选 + 业务状态仲裁（如未登录禁止查余额 Tool）
- 低置信度走澄清 / 兜底
- Checkpoint 回滚 + 重规划
- **生产最大坑不是不会做，是第一步走错**

---

### 3.4 幻觉全链路治理

| 阶段 | 手段 |
|------|------|
| 前置 | 低 temperature、规则 Prompt、少样本 |
| 过程 | 强制 RAG 溯源、工具留痕、CoT |
| 后置 | 双模型核查、Self-Refine、高风险 HITL |
| 迭代 | Bad Case 回流 → 知识库/Prompt/检索策略 |

**压力题**（豆包总结）：线上幻觉批量爆发，你作为负责人 24h 内做什么？  
→ 先止血（降级/关 Tool/切保守 Prompt）→ 对照 golden set 定位是模型/Prompt/检索/知识库 → 灰度修复 → 长期：评测门禁 + 知识库 SLA。

---

### 3.5 RAG 疑难杂症排查手册

| 症状 | 逐层排查 |
|------|----------|
| **召回率低** | 切片粒度 → Embedding 适配 → BM25 词典 → Top-K/阈值 → 索引是否更新 → 权限过滤误杀 |
| **忠诚度高但答非所问** | Query 理解差 → 加改写/拆解 |
| **TTFT 高** | 检索并行化 → 减少 Rerank 条数 → Embedding 缓存 → 小模型路由 |
| **PDF/双栏/表格** | Unstructured/OCR → 表格单独解析 → 层级切块 |
| **文档更新不同步** | 版本号 + 增量索引 + 缓存失效策略 |

---

### 3.6 效果波动：Prompt 问题还是模型问题？

**字节/腾讯方法论**：
- 固定模型换 Prompt、固定 Prompt 换模型 → 对比 golden set
- Prompt 问题：格式乱、风格飘；模型问题：推理/事实性大幅下滑
- 结合日志看召回片段、tool 入参、结构化输出解析失败率

### 3.8 系统设计压轴题

1. **千万级文档 RAG**（阿里/腾讯/国外）：分片索引、混合检索、缓存、流式 SSE、多租户、监控
2. **大模型推理平台**（字节）：多模型路由、限流熔断、灰度、Continuous Batching
3. **Agent 任务编排系统**：DAG、HITL 审批、失败重试、状态持久化
4. **AI Gateway 信用额度**（OpenAI 风格）：高并发扣减、分布式锁、按天刷新
5. **电商 AI 运营助手**（豆包）：单/多 Agent 选型、RAG 边界、数据准确性、成本

## 四、资深推演题（大厂深度思考向）

> 以下题目由面经规律 + 40K 档要求推演，**建议每个准备 5 分钟口述架构图**。

### 4.1 架构与自研

1. **抛弃 LangChain，自研 Agent 调度引擎最少要实现哪 5 个模块？**  
   → Loop 编排、Tool Registry、State Store、Trace、Policy Engine（权限/步数/审批）

2. **Workflow 和 Agent 混排时，谁拥有最终执行权？**  
   → LLM 只产出决策建议；执行层代码 + 策略引擎做最终 commit（腾讯&字节共识）

3. **100 个租户共享一套 RAG 索引，如何隔离与计费？**  
   → metadata 租户标签 + 检索 filter + 独立 rerank 配额 + 审计日志

4. **Prompt 版本如何像代码一样管理？**  
   → Git 版本 + golden set 回归 + 灰度 + 回滚；禁止线上手改

### 4.2 可靠性与成本

5. **SSE 流式中断时，如何防止重复计费与重复 Tool 执行？**  
   → 请求 idempotency key + 服务端 generation 状态机 + 客户端断线重连只读

6. **Agent 链路 P99 延迟 30s，如何定位？**  
   → Trace 拆分：Router / RAG / LLM TTFT / Tool / Rerank / Retry；看 P99 贡献占比

7. **Token 成本占毛利 40%，你会砍哪里？**  
   → 小模型路由、Prompt 压缩、Rerank 减量、缓存、批处理、减少 Planner 轮次

8. **Multi-Agent 成本翻倍，效果只提升 10%，还要不要上？**  
   → 用业务指标（转化率/人工替代率）算 ROI；能 Workflow 就不 Multi-Agent

### 4.3 安全与合规

9. **模型已 RLHF 对齐，应用层为什么还要五层安全？**  
   → 概率安全 ≠ 系统安全；Prompt 注入、Tool 越权、跨会话污染、动态权限（腾讯&字节原题）

10. **发现用户通过 Prompt 绕过护栏导出敏感数据，24h 内修复顺序？**  
    → 应用层拦截（最快）→ Tool 权限收紧 → 输出 DLP → 模型层（最慢）

11. **哪些操作必须 HITL？**  
    → 不可逆、资金、删数据、生产变更、批量外发、权限变更

### 4.4 前沿与判断

12. **Agentic RAG vs 传统 RAG：什么时候值得加 Agent？**  
    → Query 需多跳/需自检/检索失败需改写时；简单 FAQ 不值得

13. **Graph RAG 成本很高，什么情况下必须上？**  
    → 多跳推理、实体关系强、Chunk 相似度失效（法律/医疗/企业知识图谱）

14. **MCP 会替代 Function Call 吗？**  
    → 协议层标准化 vs 模型能力层；长期共存，MCP 解决工具生态互操作

15. **CLI 能替代 MCP 吗？**（淘宝原题）  
    → 不能；CLI 缺上下文调度、多源融合、长对话记忆

16. **OpenClaw 为什么火？Claude Code 为什么更复杂？**  
    → 轻量 ReAct 降低门槛 vs 生产级记忆/权限/Trace/多轮任务完整性

### 4.5 行为与思维（国外大厂 / 终面）

17. **讲一个过度设计导致延误的项目**（Anthropic 风格）
18. **AI Agent 落地最大瓶颈是什么？未来 1–2 年工程重点？**
19. **你如何看待「Harness 比模型更关键」？结合项目举例**
20. **如果团队只能投一个方向：自研框架 / 模型微调 / RAG 优化 / 评测平台，你怎么排优先级？**

## 五、疑难杂症 → 排查思路速查

| 疑难杂症 | 第一反应 | 深挖方向 | 长期根治 |
|----------|----------|----------|----------|
| 用户投诉「胡说八道」 | 查该 case 召回片段 | 检索空/无关 → RAG；有片段仍错 → 模型/Prompt | 溯源强制 + 评测门禁 |
| Agent 卡死不动 | 看最后一步 Tool/LLM | 超时未设 / 死循环 / 等人工 | 步数上限 + 心跳 + 自动降级 |
| 同一问题今天对明天错 | 对照模型/Prompt 版本 | 温度/检索随机性/索引更新 | 固定 seed + 版本锁定 + golden 回归 |
| 召回「看起来相关但业务错」 | 看 Rerank 前后 Top5 | 语义近但业务无关 | 加 metadata 过滤 + 业务 Rerank 样本 |
| JSON 输出格式崩 | 统计解析失败率 | Prompt 过长 / 模型弱 / 缺 Schema | Function Call + 后置校验重试 |
| 知识库更新后仍答旧内容 | 查索引与缓存 TTL | 向量未重建 / 缓存未失效 | 版本号 + 主动失效 + 增量索引 |
| 多 Agent 互相踢皮球 | 查消息协议与终止条件 | 无仲裁 / 无 max_round | Manager 终审 + 结构化输出 |
| QPS 高时延迟爆炸 | 看 GPU/检索/LLM 队列 | 缺 batching / 缺限流 | 异步队列 + 小模型分流 |
| 提示注入成功案例 | 查攻击路径 | 直接指令 / 间接 via 文档 | 输入隔离 + 输出 DLP + Tool 最小权限 |
