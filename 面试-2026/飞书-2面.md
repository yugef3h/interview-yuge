## 一、合同审查系统相关
### 1. 合同审查的整体架构与完整流程
**整体架构**：分为文件接入层、文本解析层、预处理层、检索层、大模型推理层、结果输出层与管控层。
**完整流程**：
用户上传合同文件（PDF/Word/图片） → 文件格式解析与文本提取 → 文本清洗、分段、结构化 → 向量化入库构建知识库 → 输入审查指令/问题 → 检索相关合同条款 → 上下文拼接送入大模型 → 模型执行风险识别/条款解读/合规校验 → 输出审查报告、风险点、修改建议 → 结果结构化展示与日志留存。

## 二、RAG（检索增强生成）相关
### 2. RAG 的完整流程
1.  **文档加载**：读取合同、法规、判例等原始文件；
2.  **文本预处理**：去噪、分段、去重、关键词提取；
3.  **向量化**：通过 Embedding 模型将文本转为向量；
4.  **向量入库**：存入向量数据库，建立索引；
5.  **用户 Query 向量化**：将问题转为同一空间向量；
6.  **检索召回**：向量检索 + 关键词检索召回相关片段；
7.  **重排精筛**：通过交叉编码器或相似度排序过滤噪声；
8.  **上下文构建**：将召回片段与问题拼接为 Prompt；
9.  **大模型推理**：基于检索内容生成回答；
10. **结果后处理**：格式整理、去重、合规校验。

### 8. 如何使用 LangGraph 构建 RAG 系统？
LangGraph 构建 RAG 采用**状态化、可中断、可循环的智能体流程**：
1.  **定义状态**：存储文档、检索结果、对话历史、模型输出；
2.  **节点拆分**：文件加载 → 文本切分 → 向量化 → 检索 → 重排 → 推理 → 输出；
3.  **条件边**：判断是否需要多轮检索/自我修正。

**文件处理方式**：支持 PDF/Word/txt/md，用 PyPDF/Unstructured 解析，递归按标题/段落分段，控制块大小与重叠字符。
**检索逻辑**：
- 向量检索：召回 Top-K 相关片段；
- 关键词检索（BM25）补充召回；
- 混合结果送入重排模型（Reranker）精排；
- 最终选取高相关片段送入大模型。

### 9. RAG 系统搭建的主要难点
1.  **长文本处理**：合同条款冗长，分段不当导致上下文断裂；
2.  **复杂逻辑问题**：多条款关联、跨章节推理能力不足；
3.  **检索不精准**：召回无关内容，引发模型幻觉；
4.  **文本截断**：Token 超限丢失关键信息；
5.  **显存溢出**：Embedding + 大模型推理占用高；
6.  **多轮对话理解**：历史上下文管理困难；
7.  **领域专业性强**：通用 Embedding 对合同术语表达弱。

### 12. 如何对 RAG 系统进行评估？
采用**人工评估 + 自动化指标结合**的方式：
1.  **检索层**：计算召回率、精确率、MRR，检验条款是否被正确召回；
2.  **生成层**：使用 RAGAS 体系，包括忠实率、答案相关性、上下文精准度、上下文召回率；
3.  **业务指标**：风险点漏检率、错误解读率、格式合规度；
4.  **人工评估**：从准确性、完整性、专业性、可读性打分；
5.  **长期优化**：通过 A/B 测试对比不同 Embedding、分段策略、重排模型效果。

## 三、模型微调相关
### 3. 指令微调中，三轮微调的实施、损失函数选择、混淆矩阵应用与数据不均衡处理
**三轮微调实施（渐进式策略）**：
1.  **第一轮：基础指令对齐**：用通用合同数据做全参数或 LoRA 微调，学习基本格式与意图；
2.  **第二轮：领域专项微调**：聚焦风险点、条款类型、合规判断，强化专业能力；
3.  **第三轮：对齐人类偏好**：人工标注优质输出做 DPO/RLHF，提升准确性与实用性。

**损失函数选择**：
- 分类任务：CrossEntropyLoss；
- 回归任务：MSE；
- 生成任务：NLLoss/CrossEntropy；
- 偏好对齐：DPO Loss。

**混淆矩阵应用**：用于分析精确率、召回率、F1，定位模型易混淆的风险类别与条款类型，指导数据补充与阈值调整。

**数据不均衡处理**：采用类别加权 Loss、过采样少数类、欠采样多数类、合成样本（SMOTE）、Focal Loss。

### 10. 模型微调时选择 Llama 2 的原因
1.  **开源可用**：可商用版本降低落地成本；
2.  **效果均衡**：7B/13B 参数适合私有化部署；
3.  **指令遵循能力强**：经过 SFT 微调，适合合同审查等任务；
4.  **生态成熟**：支持 LoRA/QLoRA 高效微调，降低硬件要求；
5.  **长文本支持较好**：能处理合同类长文本理解；
6.  **社区资源丰富**：便于二次优化与问题排查。

## 四、机器学习相关（XGBoost、LightGBM）
### 4. 数据不均衡二分类任务的评价指标及 AUC 的作用
**常用评价指标**：精确率（Precision）、召回率（Recall）、F1 分数、Fβ 分数、AUC、PR 曲线、Kappa 系数。
**AUC 的作用**：衡量模型对正负样本的区分能力，不受类别不均衡影响，能客观反映模型整体排序能力。在合同风险识别等场景中，AUC 越高，说明模型越能稳定把高风险样本排在前面，避免漏判重要风险。

### 5. XGBoost 与 LightGBM 如何加权融合？
**加权融合思路**：基于单模型表现分配权重，对预测概率做加权求和。
**操作方法**：
1.  分别训练 XGBoost 与 LightGBM，在验证集输出预测概率；
2.  根据验证集 AUC/F1 分配权重，或用网格搜索/线性回归优化权重；
3.  融合公式：`最终概率 = w1 * xgb_prob + w2 * lgb_prob`；
4.  也可采用堆叠融合（Stacking），以单模型输出作为特征，训练 LR/逻辑回归做第二层融合；
5.  按融合后设置的最优阈值输出类别。

### 6. XGBoost 定义、核心优势、学习率设置、层级设置与 LightGBM 叶子生长策略
**XGBoost 定义**：极端梯度提升树，基于梯度提升框架，支持并行计算、正则化与二阶泰勒展开。
**核心优势**：精度高、鲁棒性强、自带 L1/L2 正则防止过拟合、支持缺失值处理、列抽样、并行训练、可解释性强。
**学习率设置**：树模型一般初始 0.1~0.3，数据复杂时调低至 0.01~0.1，配合更多树数量保证收敛。
**层级设置与内存特点**：XGBoost 采用按层生长，内存占用较高，需存储大量节点特征与梯度信息。
**LightGBM 叶子生长策略**：采用按叶子生长，优先分裂增益最大的叶子，收敛更快、精度更高，同时通过直方图优化降低内存占用，适合大规模数据。

### 7. XGBoost 与 LightGBM 融合的关键因素
1.  **数据一致性**：确保训练集、验证集、特征工程完全一致；
2.  **模型差异性**：保留单模型结构差异，避免同质化导致融合增益低；
3.  **权重分配依据**：基于验证集 AUC/F1/损失分配权重，避免主观赋值；
4.  **阈值选择**：不均衡数据下需重新确定分类阈值；
5.  **推理性能**：融合后延迟是否满足上线要求；
6.  **过融合风险**：避免在训练集上过拟合，需用独立测试集验证。

## 五、工程与问题处理相关
### 13. 模型幻觉问题的解决方法
1.  **Prompt 优化**：强制要求引用原文条款，禁止无依据推断；
2.  **RAG 增强**：提升检索精准度，增加重排环节，过滤无关上下文；
3.  **思维链引导**：使用 CoT 分步推理，避免跳跃式结论；
4.  **模型微调**：用高质量合同审查数据做 SFT，减少虚构内容；
5.  **校验机制**：增加事实校验节点，对比生成内容与原文一致性；
6.  **少样本示例**：在 Prompt 中加入正确示例，规范输出格式与依据。

### 14. 数据缺失、处理上限或格式不完整的处理方法
1.  **缺失值**：数值型用均值/中位数填充，类别型用默认值，文本型标记缺失；
2.  **处理上限**：做分批处理、流式读取，超长文本自动分段，避免内存溢出；
3.  **格式不完整**：非 JSON 格式做正则提取关键信息，无法解析则丢弃并记录日志；
4.  **异常兜底**：设置默认返回结构，保证接口不崩溃；
5.  **数据校验**：前置校验模块，提前过滤脏数据，提升系统稳定性。

### 15. 多语言及中英文混合合同处理的改动点与调整思路
**需改动点**：
1.  Embedding 模型替换为多语言向量模型，支持中英语义对齐；
2.  大模型选择多语言预训练模型，增强跨语言理解；
3.  分词器适配多语言，避免中英文混合乱切分；
4.  检索与重排支持中英混合查询。

**切分调整思路**：
1.  按语言分段，中英文段落分开处理；
2.  保留标点与格式特征，避免破坏条款结构；
3.  适当降低块长度，提升混合文本检索精度；
4.  对数字、条款编号做特殊保留，不做无意义切分。
